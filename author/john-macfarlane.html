Описание

Pandoc - это библиотека Haskell для преобразования из одного формата разметки в другой и инструмент командной строки, который использует эту библиотеку.

Pandoc может конвертировать между многочисленными форматами разметки и обработки текста, включая, помимо прочего, различные разновидности Markdown , HTML , LaTeX и Word docx . Для полных списков входных и выходных форматов, см --fromи --to опции ниже . Pandoc также может создавать PDF-файлы : см. Создание PDF-файла ниже.

Расширенная версия Markdown от Pandoc включает синтаксис для таблиц , списков определений , блоков метаданных , сносок , цитат , математических расчетов и многого другого. См. Ниже в разделе « Уценка Pandoc» .

Pandoc имеет модульную структуру: он состоит из набора читателей, которые анализируют текст в заданном формате и создают собственное представление документа ( абстрактное синтаксическое дерево или AST), и набора средств записи, которые преобразуют это собственное представление в целевой формат. Таким образом, для добавления формата ввода или вывода требуется только добавить средство чтения или записи. Пользователи также могут запускать настраиваемые фильтры pandoc для изменения промежуточного AST.

Поскольку промежуточное представление документа pandoc менее выразительно, чем многие форматы, между которыми он конвертирует, не следует ожидать идеального преобразования между каждым форматом и любым другим. Pandoc пытается сохранить структурные элементы документа, но не детали форматирования, такие как размер полей. А некоторые элементы документа, такие как сложные таблицы, могут не вписываться в простую модель документа pandoc. Хотя преобразования из Markdown pandoc во все форматы стремятся быть идеальными, преобразования из форматов более выразительных, чем Markdown pandoc, можно ожидать с потерями.

Использование pandoc

Если нет ввода-файлы не указаны, вход считывается из стандартного ввода . По умолчанию вывод идет на стандартный вывод . Для вывода в файл используйте -oопцию:

pandoc -o output.html input.txt

По умолчанию pandoc создает фрагмент документа. Чтобы создать автономный документ (например, действительный файл HTML, включая <head>и <body>), используйте флаг -sили --standalone:

pandoc -s -o output.html input.txt

Для получения дополнительной информации о том, как создаются автономные документы, см. Шаблоны ниже.

Если дано несколько входных файлов, pandocони все объединятся (с пустыми строками между ними) перед синтаксическим анализом. (Используйте --file-scopeдля индивидуального анализа файлов.)

Указание форматов

Формат ввода и вывода можно указать явно с помощью параметров командной строки. Формат ввода можно указать с помощью -f/--fromпараметра, формат вывода с помощью -t/--toпараметра. Таким образом, чтобы преобразовать hello.txtMarkdown в LaTeX, вы можете ввести:

pandoc -f markdown -t latex hello.txt

Чтобы преобразовать hello.htmlHTML в Markdown:

pandoc -f html -t markdown hello.html

Поддерживаемые форматы ввода и вывода перечислены ниже в разделе « Параметры» (см. -fФорматы ввода и форматы -tвывода). Вы также можете использовать pandoc --list-input-formatsи pandoc --list-output-formatsдля печати списков поддерживаемых форматов.

Если формат ввода или вывода не указан явно, pandocбудет пытаться угадать его по расширениям имен файлов. Так, например,

pandoc -o hello.tex hello.txt

преобразует hello.txtMarkdown в LaTeX. Если выходной файл не указан (так что вывод идет на стандартный вывод ) или если расширение выходного файла неизвестно, формат вывода по умолчанию будет HTML. Если входной файл не задан (так что вход поступает из стандартного ввода ), или если расширения входных файлов неизвестны, формат ввода будет считать уценки.

Кодировка символов

Pandoc использует кодировку символов UTF-8 как для ввода, так и для вывода. Если ваша локальная кодировка символов не UTF-8, вы должны направлять ввод и вывод по конвейеру через iconv:

iconv -t utf-8 input.txt | pandoc | iconv -f utf-8

Обратите внимание, что в некоторых форматах вывода (таких как HTML, LaTeX, ConTeXt, RTF, OPML, DocBook и Texinfo) информация о кодировке символов включается в заголовок документа, который будет включен только при использовании этой -s/--standaloneопции.

Создание PDF

Чтобы создать PDF-файл, укажите выходной файл с .pdfрасширением:

pandoc test.txt -o test.pdf

По умолчанию pandoc будет использовать LaTeX для создания PDF-файла, для чего требуется установка движка LaTeX (см. --pdf-engineНиже). В качестве альтернативы pandoc может использовать ConTeXt, roff ms или HTML в качестве промежуточного формата. Для этого укажите выходной файл с .pdfрасширением, как и раньше, но добавить --pdf-engineпараметр или -t context, -t htmlили -t msв командной строке. Инструмент, используемый для создания PDF-файла из промежуточного формата, можно указать с помощью --pdf-engine.

Вы можете управлять стилем PDF с помощью переменных, в зависимости от используемого промежуточного формата: см. Переменные для LaTeX , переменные для ConTeXt , переменные дляwkhtmltopdf , переменные для ms . Когда HTML используется в качестве промежуточного формата, для вывода можно использовать стили --css.

Для отладки создания PDF может быть полезно посмотреть на промежуточное представление: вместо -o test.pdf, например, -s -o test.texдля вывода сгенерированного LaTeX. Затем вы можете проверить это с помощью pdflatex test.tex.

При использовании LaTeX следующие пакеты должны быть доступны (они включены все последние версии TeX Live ): amsfonts, amsmath, lm, unicode-math, ifxetex, ifluatex, listings(если --listingsиспользуется опция), fancyvrb, longtable, booktabs, graphicx(если документ содержит изображения) hyperref, xcolor, ulem, geometry(с geometryнабором переменных), setspace(с linestretch) и babel(с lang). Использование xelatexили, lualatexкак того требует движок PDF fontspec. xelatexиспользует polyglossia(с lang), xecjkи bidi(с dirнабором переменных). Если mathspecпеременная установлена, xelatexбудет использоватьсяmathspecвместо unicode-math. upquoteИ microtypeпакеты используются при наличии, и csquotesбудет использоваться для книгопечатания , если csquotesпеременная или метаданных поле установлено в истинное значение. natbib, biblatex, bibtex, И biberпакеты могут быть дополнительно использованы для цитирования рендеринга . Следующие пакеты будут использоваться для улучшения качества вывода, если они присутствуют, но pandoc не требует их присутствия: upquote(для прямых кавычек в дословной среде), microtype(для лучшей корректировки интервалов), parskip(для лучшего межпунктового пространства), xurl(для лучшие разрывы строк в URL-адресах), bookmark(для улучшения закладок PDF) и footnotehyperили footnote(чтобы разрешить сноски в таблицах).

Чтение из Интернета

Вместо входного файла может быть указан абсолютный URI. В этом случае pandoc будет получать контент с помощью HTTP:

pandoc -f html -t markdown https://www.fsf.org

При запросе документа по URL-адресу можно указать настраиваемую строку User-Agent или другой заголовок:

pandoc -f html -t markdown --request-header User-Agent:"Mozilla/5.0" \
  https://www.fsf.org

Варианты чтения

--shift-heading-level-by=ЧИСЛО

Сдвиг уровня заголовка на положительное или отрицательное целое число. Например, с --shift-heading-level-by=-1заголовки уровня 2 становятся заголовками уровня 1, а заголовки уровня 3 становятся заголовками уровня 2. Заголовки не могут иметь уровень ниже 1, поэтому заголовок, который будет смещен ниже уровня 1, становится обычным абзацем. Исключение: при сдвиге -N заголовок уровня N в начале документа заменяет заголовок метаданных. --shift-heading-level-by=-1является хорошим выбором при преобразовании документов HTML или Markdown, в которых используется начальный заголовок уровня 1 для заголовка документа и заголовки уровня 2 + для разделов. --shift-heading-level-by=1может быть хорошим выбором для преобразования документов Markdown, которые используют заголовки уровня 1 для разделов, в HTML, поскольку pandoc использует заголовок уровня 1 для отображения заголовка документа.

--base-header-level=ЧИСЛО

Устаревшее. --shift-heading-level-byВместо этого используйте = X, где X = ЧИСЛО - 1. Укажите базовый уровень для заголовков (по умолчанию 1).

--strip-empty-paragraphs

Устаревшее. +empty_paragraphsВместо этого используйте расширение. Игнорировать абзацы без содержания. Этот параметр полезен для преобразования документов текстового редактора, в которых пользователи использовали пустые абзацы для создания межпунктового пространства.

--indented-code-classes=КЛАССЫ

Укажите классы, которые будут использоваться для блоков кода с отступом, например, perl,numberLinesили haskell. Несколько классов можно разделять пробелами или запятыми.

--default-image-extension=РАСПРОСТРАНЕНИЕ

Укажите расширение по умолчанию, которое будет использоваться, когда пути к изображениям / URL-адреса не имеют расширения. Это позволяет использовать один и тот же источник для форматов, требующих разных типов изображений. В настоящее время эта опция влияет только на считыватели Markdown и LaTeX.

--file-scope

Анализируйте каждый файл отдельно перед объединением для многофайловых документов. Это позволит сноскам в разных файлах с одинаковыми идентификаторами работать должным образом. Если этот параметр установлен, сноски и ссылки не будут работать в файлах. Чтение бинарных файлов (docx, odt, epub) подразумевает --file-scope.

-F ПРОГРАММА , --filter=ПРОГРАММА

Укажите исполняемый файл, который будет использоваться в качестве фильтра, преобразующего AST pandoc после синтаксического анализа ввода и до записи вывода. Исполняемый файл должен читать JSON из stdin и записывать JSON в stdout. JSON должен быть отформатирован как собственный ввод и вывод JSON pandoc. Имя выходного формата будет передано фильтру в качестве первого аргумента. Следовательно,

pandoc --filter ./caps.py -t latex

эквивалентно

pandoc -t json | ./caps.py latex | pandoc -f json -t latex

Последняя форма может быть полезна для отладки фильтров.

Фильтры могут быть написаны на любом языке. Text.Pandoc.JSONэкспорт toJSONFilterдля облегчения написания фильтров в Haskell. Те, кто предпочитает писать фильтры на python, могут использовать модуль pandocfilters, устанавливаемый из PyPI. Есть также pandoc фильтр библиотеки PHP , Perl и JavaScript / Node.js .

В порядке предпочтения pandoc будет искать фильтры в

  1. указанный полный или относительный путь (исполняемый или неисполняемый)

  2. $DATADIR/filters(исполняемый или неисполняемый), где $DATADIR- каталог пользовательских данных (см. --data-dirвыше).

  3. $PATH (только исполняемый файл)

Фильтры и Lua-фильтры применяются в порядке, указанном в командной строке.

-L СКРИПТ , --lua-filter=СКРИПТ

Преобразуйте документ аналогично фильтрам JSON (см. --filterРаздел "Ресурсы"), но используйте встроенную систему фильтрации Lua от pandoc. Ожидается, что данный сценарий Lua вернет список фильтров Lua, которые будут применяться по порядку. Каждый фильтр Lua должен содержать функции преобразования элементов, индексированные по имени элемента AST, к которому должна применяться функция фильтра.

Модуль pandocLua предоставляет вспомогательные функции для создания элементов. Он всегда загружается в среду Lua сценария.

Ниже приведен пример сценария Lua для расширения макросов:

function expand_hello_world(inline)
  if inline.c == '{{helloworld}}' then
    return pandoc.Emph{ pandoc.Str "Hello, World" }
  else
    return inline
  end
end

return {{Str = expand_hello_world}}

В порядке предпочтения pandoc будет искать фильтры Lua в

  1. указанный полный или относительный путь (исполняемый или неисполняемый)

  2. $DATADIR/filters(исполняемый или неисполняемый), где $DATADIR- каталог пользовательских данных (см. --data-dirвыше).

-M КЛЮЧ [ =VAL ], --metadata=КЛЮЧ [ :VAL ]

Установите для поля метаданных KEY значение VAL . Значение, указанное в командной строке, переопределяет значение, указанное в документе с использованием блоков метаданных YAML . Значения будут анализироваться как логические или строковые значения YAML. Если значение не указано, значение будет рассматриваться как логическое истинное. Например --variable, --metadataвызывает установку переменных шаблона. Но, в отличие от --variable, --metadataвлияет на метаданные базового документа (который доступен из фильтров и может быть напечатан в некоторых выходных форматах), а значения метаданных будут экранированы при вставке в шаблон.

--metadata-file=ФАЙЛ

Прочтите метаданные из предоставленного файла YAML (или JSON). Этот параметр можно использовать с любым входным форматом, но скаляры строк в файле YAML всегда будут анализироваться как Markdown. Обычно ввод обрабатывается так же, как и в блоках метаданных YAML . Эту опцию можно использовать многократно для включения нескольких файлов метаданных; значения в файлах, указанных позже в командной строке, будут предпочтительнее значений, указанных в более ранних файлах. Значения метаданных, указанные внутри документа, или с помощью -M, перезаписывают значения, указанные с помощью этой опции.

-p, --preserve-tabs

Сохраняйте табуляции вместо того, чтобы преобразовывать их в пробелы. (По умолчанию pandoc преобразует табуляции в пробелы перед синтаксическим анализом своего ввода.) Обратите внимание, что это повлияет только на табуляции в буквальных диапазонах кода и блоках кода. Табуляторы в обычном тексте всегда рассматриваются как пробелы.

--tab-stop=ЧИСЛО

Укажите количество пробелов на вкладке (по умолчанию 4).

--track-changes=accept| reject|all

Определяет, что делать со вставками, удалениями и комментариями, созданными функцией MS Word «Отслеживать изменения». accept(по умолчанию), вставляет все вставки и игнорирует все удаления. rejectвставляет все удаления и игнорирует вставки. Оба acceptи rejectигнорируют комментарии. allставит в вставок, удалений и комментарии, завернутые в пролетах с insertion, deletion, comment-startи comment-endклассы, соответственно. Автор и время изменения указаны. allполезен для написания сценариев: принятие изменений только от определенного рецензента, скажем, или до определенной даты. Если абзац вставлен или удален, track-changes=allсоздает диапазон с классом paragraph-insertion/ paragraph-deletionперед затронутым разрывом абзаца. Этот параметр влияет только на программу чтения docx.

--extract-media=DIR

Извлеките изображения и другие носители, содержащиеся в исходном документе или связанные из него, в путь DIR , создав его при необходимости, и настройте ссылки на изображения в документе так, чтобы они указывали на извлеченные файлы. Если исходный формат является двоичным контейнером (docx, epub или odt), носитель извлекается из контейнера и используются исходные имена файлов. В противном случае носитель считывается из файловой системы или загружается, а новые имена файлов создаются на основе хэшей SHA1 содержимого.

--abbreviations=ФАЙЛ

Задает файл настраиваемых сокращений с сокращениями по одной строке. Если этот параметр не указан, pandoc будет читать файл данных abbreviationsиз каталога пользовательских данных или вернуться к системным значениям по умолчанию. Чтобы увидеть систему по умолчанию, используйте pandoc --print-default-data-file=abbreviations. Этот список pandoc использует только в программе чтения Markdown. Строки, заканчивающиеся точкой, которые находятся в этом списке, будут сопровождаться неразрывным пробелом, так что точка не будет создавать пробел в конце предложения в таких форматах, как LaTeX.

Общие параметры писателя

-s, --standalone

Создавайте выходные данные с соответствующим верхним и нижним колонтитулами (например, автономный файл HTML, LaTeX, TEI или RTF, а не фрагмент). Эта опция устанавливается автоматически pdf, epub, epub3, fb2, docx, и odtвыход. Для nativeвывода эта опция вызывает включение метаданных; в противном случае метаданные подавляются.

--template=ФАЙЛ | URL

Использовать указанный файл в качестве настраиваемого шаблона для созданного документа. Подразумевает --standalone. Описание синтаксиса шаблона см. В разделе « Шаблоны» ниже. Если не указано расширение, будет добавлено расширение , соответствующее писателю, так что --template=specialвнешний вид для special.htmlдля HTML вывода. Если шаблон не найден, pandoc будет искать его в templatesподкаталоге каталога пользовательских данных (см. --data-dir). Если этот параметр не используется, будет использоваться шаблон по умолчанию, соответствующий формату вывода (см. -D/--print-default-template).

-V КЛЮЧ [ =VAL ], --variable=КЛЮЧ [ :VAL ]

Установите для переменной шаблона KEY значение VAL при рендеринге документа в автономном режиме. Если VAL не указан, ключу будет присвоено значение true.

-D ФОРМАТ , --print-default-template=ФОРМАТ

Распечатайте системный шаблон по умолчанию для выходного ФОРМАТА . (См. -tСписок возможных форматов FORMAT .) Шаблоны в каталоге пользовательских данных игнорируются. Этот параметр можно использовать с -o/ --outputдля перенаправления вывода в файл, но -o/ --outputдолжен стоять раньше --print-default-templateв командной строке.

Обратите внимание, что в некоторых шаблонах по умолчанию, например, используются партиалы styles.html. Для печати партиалы, используйте --print-default-data-file, например, --print-default-data-file=templates/styles.html.

--print-default-data-file=ФАЙЛ

Распечатайте системный файл данных по умолчанию. Файлы в каталоге пользовательских данных игнорируются. Этот параметр можно использовать с -o/ --outputдля перенаправления вывода в файл, но -o/ --outputдолжен стоять раньше --print-default-data-fileв командной строке.

--eol=crlf| lf|native

Вручную укажите окончания строк: crlf(Windows), lf(macOS / Linux / UNIX) или native(окончания строк, соответствующие ОС, в которой запущен pandoc). По умолчанию это native.

--dpi= ЧИСЛО

Укажите значение dpi (точек на дюйм) по умолчанию для преобразования из пикселей в дюймы / сантиметры и наоборот. (Технически правильным термином будет ppi: пиксели на дюйм). По умолчанию 96 точек на дюйм. Когда изображения содержат внутреннюю информацию о точках на дюйм, используется закодированное значение вместо значения по умолчанию, заданного этой опцией.

--wrap=auto| none|preserve

Определите, как текст помещается в выходные данные (исходный код, а не визуализированная версия). С auto(по умолчанию) pandoc будет пытаться обернуть строки до ширины столбца, указанной в --columns(по умолчанию 72). С nonepandoc вообще не будет переносить строки. С preservepandoc попытается сохранить упаковку из исходного документа (то есть, если в исходном тексте есть несемантические символы новой строки, в выводе также будут несемантические символы новой строки). Автоматическая упаковка в настоящее время не работает при выводе HTML. В ipynbвыводе этот параметр влияет на перенос содержимого ячеек уценки.

--columns=ЧИСЛО

Укажите длину строк в символах. Это влияет на перенос текста в сгенерированном исходном коде (см. --wrap). Это также влияет на расчет ширины столбцов для текстовых таблиц (см. Таблицы ниже).

--toc, --table-of-contents

Включите автоматически сгенерированного оглавления (или, в случае latex, context, docx, odt, opendocument, rst, или ms, инструкция , чтобы создать одну) в выходном документе. Этот параметр не имеет никакого эффекта , если -s/--standaloneиспользуется, и оно не имеет никакого влияния на man, docbook4, docbook5или jatsвыход.

Обратите внимание, что если вы создаете PDF-файл через ms, оглавление появится в начале документа перед заголовком. Если вы предпочитаете, чтобы он находился в конце документа, воспользуйтесь опцией --pdf-engine-opt=--no-toc-relocation.

--toc-depth=ЧИСЛО

Укажите количество уровней раздела для включения в оглавление. По умолчанию - 3 (это означает, что заголовки уровня 1, 2 и 3 будут перечислены в содержимом).

--strip-comments

Удалите комментарии HTML в исходном коде Markdown или Textile, а не передавайте их в Markdown, Textile или HTML как необработанный HTML. Это не относится к комментариям HTML внутри необработанных блоков HTML, если markdown_in_html_blocksрасширение не установлено.

--no-highlight

Отключает подсветку синтаксиса для блоков кода и встроенных строк, даже если указан атрибут языка.

--highlight-style=СТИЛЬ | ФАЙЛ

Задает стиль окраски, который будет использоваться в выделенном исходном коде. Варианты pygments(по умолчанию), kate, monochrome, breezeDark, espresso, zenburn, haddock, и tango. Для получения дополнительной информации о выделении синтаксиса в pandoc см. Раздел « Подсветка синтаксиса» ниже. См. Также --list-highlight-styles.

Вместо имени СТИЛЬ.theme может быть предоставлен файл JSON с расширением . Это будет проанализировано как тема выделения синтаксиса KDE и (если допустимо) будет использоваться как стиль выделения.

Чтобы сгенерировать JSON-версию существующего стиля, используйте --print-highlight-style.

--print-highlight-style=СТИЛЬ | ФАЙЛ

Печатает версию стиля выделения в формате JSON, которую можно изменить, сохранить с .themeрасширением и использовать с --highlight-style. Этот параметр можно использовать с -o/ --outputдля перенаправления вывода в файл, но -o/ --outputдолжен стоять раньше --print-highlight-styleв командной строке.

--syntax-definition=ФАЙЛ

Указывает pandoc загрузить файл определения синтаксиса KDE XML, который будет использоваться для выделения синтаксиса правильно отмеченных блоков кода. Это можно использовать для добавления поддержки новых языков или использования измененных определений синтаксиса для существующих языков. Эту опцию можно повторить, чтобы добавить несколько определений синтаксиса.

-H ФАЙЛ , --include-in-header=ФАЙЛ | URL

Включите содержимое FILE дословно в конец заголовка. Это можно использовать, например, для включения специального CSS или JavaScript в HTML-документы. Эту опцию можно использовать многократно для включения нескольких файлов в заголовок. Они будут включены в указанном порядке. Подразумевает --standalone.

-B ФАЙЛ , --include-before-body=ФАЙЛ | URL

Включите содержимое FILE дословно в начало тела документа (например, после <body>тега в HTML или \begin{document}команды в LaTeX). Это можно использовать для включения панелей навигации или баннеров в документы HTML. Эту опцию можно использовать многократно для включения нескольких файлов. Они будут включены в указанном порядке. Подразумевает --standalone.

-A ФАЙЛ , --include-after-body=ФАЙЛ | URL

Включите содержимое FILE дословно в конец тела документа (перед </body>тегом в HTML или \end{document}командой в LaTeX). Эту опцию можно использовать многократно для включения нескольких файлов. Они будут включены в указанном порядке. Подразумевает --standalone.

--resource-path=SearchPath

Список путей для поиска изображений и других ресурсов. Пути должны быть разделены :в системах Linux, UNIX и macOS и ;в Windows. Если --resource-pathне указан, путь к ресурсу по умолчанию - это рабочий каталог. Обратите внимание, что, если --resource-pathуказано, рабочий каталог должен быть явно указан, иначе поиск в нем не будет выполняться. Например: --resource-path=.:testбудет искать в рабочем каталоге и testподкаталоге именно в таком порядке.

--resource-pathтолько имеет эффект , если (а) формат выходного изображения встраивает (например, docx, pdf, или htmlс --self-contained) или (б) он используется вместе с --extract-media.

Создайте автономный HTML-файл без внешних зависимостей, используя data:URI для включения содержимого связанных скриптов, таблиц стилей, изображений и видео. Подразумевает --standalone. Преобразование TeX математики MathML (в epub3, docbook4, docbook5, jats, html4и html5). Это значение по умолчанию для odtвывода. Обратите внимание, что в настоящее время только Firefox и Safari (и некоторые программы для чтения электронных книг) изначально поддерживают MathML.

Преобразуйте формулы TeX в <img>теги, которые ссылаются на внешний скрипт, преобразующий формулы в изображения. Формула будет закодирована в URL-адресе и объединена с предоставленным URL-адресом. --gladtex

Включайте математику TeX в <eq>теги в выводе HTML. Полученный HTML- код затем может быть обработан GladTeX для создания изображений наборных формул и HTML-файла со ссылками на эти изображения. Итак, порядок действий такой:

pandoc -s --gladtex input.md -o myfile.htex
gladtex -d myfile-images myfile.htex
# produces myfile.html and images in myfile-images

Шаблоны

Когда этот -s/--standaloneпараметр используется, pandoc использует шаблон для добавления материала верхнего и нижнего колонтитула, необходимого для самостоятельного документа. Чтобы увидеть используемый по умолчанию шаблон, просто введите

pandoc -D *FORMAT*

где FORMAT - это имя выходного формата. Пользовательский шаблон можно указать с помощью --templateопции. Вы также можете переопределить системные шаблоны по умолчанию для данного формата вывода FORMAT , поместив файл templates/default.*FORMAT*в каталог пользовательских данных (см. --data-dirВыше). Исключения:

  • Для odtвывода настройте default.opendocumentшаблон.
  • Для pdfвывода настройте default.latexшаблон (или default.contextшаблон, если вы используете -t context, или default.msшаблон, если вы используете -t ms, или default.htmlшаблон, если вы используете -t html).
  • docxи не pptxимеют шаблона (однако вы можете использовать его --reference-docдля настройки вывода).

Шаблоны содержат переменные , которые позволяют включать произвольную информацию в любой момент файла. Их можно установить в командной строке с помощью -V/--variableопции. Если переменная не установлена, pandoc будет искать ключ в метаданных документа, который может быть установлен либо с помощью блоков метаданных YAML, либо с -M/--metadataопцией. Кроме того, pandoc дает некоторым переменным значения по умолчанию. См. Переменные ниже для получения списка переменных, используемых в шаблонах по умолчанию pandoc.

Если вы используете настраиваемые шаблоны, вам может потребоваться отредактировать их при изменении pandoc. Мы рекомендуем отслеживать изменения в шаблонах по умолчанию и соответствующим образом изменять свои собственные шаблоны. Легкий способ сделать это - разветвить репозиторий pandoc-templates и объединить изменения после каждого выпуска pandoc.

Синтаксис шаблона

Комментарии

Все, что находится между последовательностью $--и концом строки, будет рассматриваться как комментарий и исключаться из вывода.

Разделители

Чтобы пометить переменные и управляющие структуры в шаблоне, в качестве разделителей можно использовать $… $или ${… }. Стили также могут быть смешаны в одном шаблоне, но открывающий и закрывающий разделители должны совпадать в каждом случае. За открывающим разделителем могут следовать один или несколько пробелов или табуляции, которые будут проигнорированы. За закрывающим разделителем могут следовать один или несколько пробелов или табуляции, которые будут проигнорированы.

Чтобы включить $в документ литерал , используйте $$.

Интерполированные переменные

Слот для интерполированной переменной - это имя переменной, окруженное совпадающими разделителями. Имена переменных должны начинаться с буквы и может содержать буквы, цифры, _, -, и .. Ключевые слова it, if, else, endif, for, sep, и endforне могут быть использованы в качестве имен переменных. Примеры:

$foo$
$foo.bar.baz$
$foo_bar.baz-bim$
$ foo $
${foo}
${foo.bar.baz}
${foo_bar.baz-bim}
${ foo }

Имена переменных с точками используются для получения значений структурированных переменных. Так, например, employee.salaryвернет значение salaryполя объекта, которое является значением employeeполя.

  • Если значение переменной является простым значением, оно будет отображаться дословно. (Обратите внимание, что экранирование не выполняется; предполагается, что вызывающая программа экранирует строки в соответствии с форматом вывода.)
  • Если значение является списком, значения будут объединены.
  • Если значением является карта, строка trueбудет отображена.
  • Все остальные значения будут отображаться как пустая строка.

Conditionals

Условное выражение начинается с if(variable)(заключено в совпадающие разделители) и заканчивается endif(заключено в совпадающие разделители). Он может необязательно содержать else(заключенный в совпадающие разделители). ifСекция используется , если variableимеет непустое значение, в противном случае elseиспользуется часть (если она присутствует). Примеры:

$if(foo)$bar$endif$

$if(foo)$
  $foo$
$endif$

$if(foo)$
part one
$else$
part two
$endif$

${if(foo)}bar${endif}

${if(foo)}
  ${foo}
${endif}

${if(foo)}
${ foo.bar }
${else}
no foo!
${endif}

Ключевое слово elseifможет использоваться для упрощения сложных вложенных условных выражений:

$if(foo)$
XXX
$elseif(bar)$
YYY
$else$
ZZZ
$endif$

Для петель

Цикл for начинается с for(variable)(заключен в совпадающие разделители) и заканчивается endfor(заключен в совпадающие разделители.

  • Если variableэто массив, материал внутри цикла будет оцениваться повторно, variableпо очереди устанавливая каждое значение массива и объединяя их.
  • Если variableэто карта, материал внутри будет установлен на карту.
  • Если значение связанной переменной не является массивом или картой, для ее значения будет выполнена одна итерация.

Примеры:

$for(foo)$$foo$$sep$, $endfor$

$for(foo)$
  - $foo.last$, $foo.first$
$endfor$

${ for(foo.bar) }
  - ${ foo.bar.last }, ${ foo.bar.first }
${ endfor }

$for(mymap)$
$it.name$: $it.office$
$endfor$

При желании вы можете указать разделитель между последовательными значениями, используя sep(заключенный в совпадающие разделители). Материал между sepи endforявляется разделителем.

${ for(foo) }${ foo }${ sep }, ${ endfor }

Вместо использования variableвнутри цикла itможно использовать специальное анафорическое ключевое слово .

${ for(foo.bar) }
  - ${ it.last }, ${ it.first }
${ endfor }

Partials

Части (подшаблоны, хранящиеся в разных файлах) могут быть включены, используя имя части, за которой следует (), например:

${ styles() }

Частичные файлы будут искать в каталоге, содержащем основной шаблон. Предполагается, что имя файла имеет то же расширение, что и основной шаблон, если у него нет расширения. При вызове партиала также может использоваться полное имя, включая расширение файла:

${ styles.html() }

(Если частичное не найдено в каталоге шаблона, его также можно будет искать в templatesподкаталоге каталога пользовательских данных.)

Частичные значения могут при желании применяться к переменным через двоеточие:

${ date:fancy() }

${ articles:bibentry() }

Если articlesэто массив, он будет перебирать его значения, применяя партиал bibentry()к каждому из них. Таким образом, второй пример выше эквивалентен

${ for(articles) }
${ it:bibentry() }
${ endfor }

Обратите внимание, что itпри итерации по частям необходимо использовать ключевое слово anaphoric. В приведенных выше примерах bibentryпартиал должен содержать it.title(и так далее) вместо articles.title.

Заключительные символы новой строки опускаются во включенных частях.

Части могут включать в себя другие части.

Разделитель между значениями массива может быть указан в квадратных скобках сразу после имени переменной или частично:

${months[, ]}$

${articles:bibentry()[; ]$

Разделитель в этом случае является буквальным и (в отличие от с sepв явном forцикле) не может содержать интерполированные переменные или другие директивы шаблона.

гнездование

Чтобы обеспечить «вложенность» содержимого, то есть с отступом последующих строк, используйте ^директиву:

$item.number$  $^$$item.description$ ($item.price$)

В этом примере, если item.descriptionимеется несколько строк, все они будут с отступом, чтобы соответствовать первой строке:

00123  A fine bottle of 18-year old
       Oban whiskey. ($148)

Чтобы вложить несколько строк на один уровень, выровняйте их по ^директиве в шаблоне. Например:

$item.number$  $^$$item.description$ ($item.price$)
               (Available til $item.sellby$.)

будет производить

00123  A fine bottle of 18-year old
       Oban whiskey. ($148)
       (Available til March 30, 2020.)

Если переменная встречается сама по себе в строке, ей предшествует пробел и не следует дальнейший текст или директивы в той же строке, а значение переменной содержит несколько строк, оно будет автоматически вложено.

Бьющиеся пространства

Обычно пробелы в самом шаблоне (в отличие от значений интерполированных переменных) не могут быть разбиты, но их можно сделать разбиваемыми в части шаблона с помощью ~ключевого слова (заканчивающегося другим ~).

$~$This long line may break if the document is rendered
with a short line length.$~$

трубы

Канал преобразует значение переменной или частичное. Каналы указываются с помощью косой черты ( /) между именем переменной (или частичным) и именем канала. Пример:

$for(name)$
$name/uppercase$
$endfor$

$for(metadata/pairs)$
- $it.key$: $it.value$
$endfor$

$employee:name()/uppercase$

Связать трубы можно:

$for(employees/pairs)$
$it.key/alpha/uppercase$. $it.name$
$endfor$

Некоторые трубы принимают параметры:

|----------------------|------------|
$for(employee)$
$it.name.first/uppercase/left 20 "| "$$it.name.salary/right 10 " | " " |"$
$endfor$
|----------------------|------------|

В настоящее время предопределены следующие каналы:

  • pairsПреобразует карту или массив в массив карт, каждая из которых keyи valueполей. Если исходное значение было массивом, то keyбудет индексом массива, начиная с 1.

  • uppercase: Переводит текст в верхний регистр.

  • lowercase: Переводит текст в нижний регистр.

  • length: Возвращает длину значения: количество символов для текстового значения, количество элементов для карты или массива.

  • reverse: Переворачивает текстовое значение или массив и не влияет на другие значения.

  • first: Возвращает первое значение массива, если оно применено к непустому массиву; в противном случае возвращает исходное значение.

  • last: Возвращает последнее значение массива, если оно применено к непустому массиву; в противном случае возвращает исходное значение.

  • rest: Возвращает все значения массива, кроме первого, если применяется к непустому массиву; в противном случае возвращает исходное значение.

  • allbutlast: Возвращает все значения массива, кроме последнего, если применяется к непустому массиву; в противном случае возвращает исходное значение.

  • chomp: Удаляет завершающие символы новой строки (и разбиваемый пробел).

  • nowrap: Отключает перенос строк на разбиваемые пробелы.

  • alpha: Преобразует текстовые значения, которые можно читать как целые числа, в буквы нижнего регистра a..z(мод 26). Это можно использовать для получения буквенного перечисления из индексов массива. Чтобы получить прописные буквы, соедините их в цепочку uppercase.

  • roman: Преобразует текстовые значения, которые можно читать как целые числа, в римские числа в нижнем регистре. Это можно использовать для получения буквенного перечисления из индексов массива. Чтобы получить римские буквы в верхнем регистре, соедините их с uppercase.

  • left n "leftborder" "rightborder": Отображает текстовое значение в блоке шириной n, выровненным по левому краю, с необязательной левой и правой границей. Не влияет на другие значения. Поведение некоторых читателей и писателей можно настроить, включив или отключив различные расширения.

Читатель и писатель уценки, безусловно, больше всего используют расширения. Расширения, используемые только ими, поэтому рассматриваются в разделе Pandoc Markdown ниже (см. Варианты Markdown для commonmarkи gfm.) Далее рассматриваются расширения, которые также работают для других форматов.

Интерпретируйте прямые кавычки как фигурные кавычки, ---длинные тире, --короткие тире и ...эллипсы. Неразрывные пробелы вставляются после определенных сокращений, таких как «Мистер».

Это расширение можно включить / отключить для следующих форматов:

форматы ввода

форматы вывода
markdown, latex, context,rst
включен по умолчанию в
markdown, latex, context(Вход и выход)

Примечание. Если вы пишете Markdown, smartрасширение дает обратный эффект: фигурные кавычки выходят прямо.

В LaTeX smartозначает использование стандартных лигатур TeX для кавычек ( ``и ''для двойных кавычек, `и 'для одинарных кавычек) и тире ( --для тире и ---длинного тире). Если smartотключено, то при чтении LaTeX pandoc будет буквально разбирать эти символы. При написании LaTeX включение smartуказывает pandoc использовать лигатуры, когда это возможно; если smartон отключен, pandoc будет использовать кавычки Unicode и символы тире.

Заголовки и разделы

Расширение: auto_identifiers

Заголовку без явно указанного идентификатора будет автоматически присвоен уникальный идентификатор на основе текста заголовка.

Это расширение можно включить / отключить для следующих форматов:

форматы ввода
markdown, latex, rst, mediawiki,textile
форматы вывода
markdown, muse
включен по умолчанию в
markdown, muse

Алгоритм по умолчанию, используемый для получения идентификатора из текста заголовка:

  • Удалите все форматирование, ссылки и т. Д.
  • Убрать все сноски.
  • Удалите все не буквенно-цифровые символы, кроме подчеркивания, дефисов и точек.
  • Замените все пробелы и символы новой строки дефисами.
  • Преобразуйте все буквенные символы в нижний регистр.
  • Удалите все, вплоть до первой буквы (идентификаторы не могут начинаться с цифры или знака препинания).
  • Если после этого ничего не осталось, используйте идентификатор section.

Так, например,

Heading Идентификатор
Heading identifiers in HTML heading-identifiers-in-html
Maître d'hôtel maître-dhôtel
*Dogs*?--in *my* house? dogs--in-my-house
[HTML], [S5], or [RTF]? html-s5-or-rtf
3. Applications applications
33 section

Эти правила в большинстве случаев должны позволять определять идентификатор по тексту заголовка. Исключение составляют случаи, когда несколько заголовков имеют одинаковый текст; в этом случае первый получит идентификатор, как описано выше; второй получит тот же идентификатор с -1добавлением; третий с -2; и так далее.

(Однако, если gfm_auto_identifiersон включен , используется другой алгоритм ; см. Ниже.)

Эти идентификаторы используются для предоставления целей ссылки в оглавлении, созданном --toc|--table-of-contentsопцией. Они также упрощают предоставление ссылок из одного раздела документа в другой. Ссылка на этот раздел, например, может выглядеть так:

See the section on
[heading identifiers](#heading-identifiers-in-html-latex-and-context).

Однако обратите внимание, что этот метод предоставления ссылок на разделы работает только в форматах HTML, LaTeX и ConTeXt.

Если --section-divsопция указана, то каждый раздел будет заключен в section(или div, если html4был указан), и идентификатор будет прикреплен к включающему тегу <section>(или <div>), а не к самому заголовку. Это позволяет управлять целыми разделами с помощью JavaScript или по-другому обрабатывать в CSS.

Расширение: ascii_identifiers

Приводит к тому, что идентификаторы, создаваемые с помощью, auto_identifiersявляются чистым ASCII. Ударения удаляются из латинских букв с ударением, а нелатинские буквы опускаются.

Расширение: gfm_auto_identifiers

Изменяет алгоритм, используемый auto_identifiersдля соответствия методу GitHub. Пробелы преобразуются в тире ( -), символы верхнего регистра - в символы нижнего регистра, а символы пунктуации, кроме -и _, удаляются. Emojis заменяются их именами.

Грамотная поддержка Haskell

Расширение: literate_haskell

Рассматривайте этот документ как грамотный исходный код Haskell.

Это расширение можно включить / отключить для следующих форматов:

форматы ввода
markdown, rst,latex
форматы вывода
markdown, rst, latex,html

Если вы добавите +lhs(или +literate_haskell) к одному из вышеперечисленных форматов, pandoc будет рассматривать документ как грамотный исходный код Haskell. Это означает, что

  • При вводе Markdown разделы «птичьего трека» будут анализироваться как код Haskell, а не как блочные цитаты. Текст между \begin{code}и \end{code}также будет рассматриваться как код Haskell. Для заголовков в стиле ATX будет использоваться символ «=» вместо «#».

  • В выводе Markdown блоки кода с классами haskellи literateбудут отображаться с использованием треков птиц, а цитаты блоков будут иметь отступ на один пробел, поэтому они не будут рассматриваться как код Haskell. Кроме того, заголовки будут отображаться в виде текстового (с подчеркиванием), а не в стиле ATX (с символами '#'). (Это связано с тем, что ghc рассматривает символы '#' в столбце 1 как вводящие номера строк.)

  • При вводе реструктурированного текста разделы «птичьего следа» будут анализироваться как код Haskell.

  • При выводе реструктурированного текста блоки кода с классом haskellбудут отображаться с использованием птичьих следов.

  • При вводе LaTeX текст в codeсредах будет анализироваться как код Haskell.

  • В выводе LaTeX блоки кода с классом haskellбудут отображаться внутри codeсред.

  • В выводе HTML блоки кода с классом haskellбудут отображаться с literatehaskellтреками классов и птиц.

Примеры:

pandoc -f markdown+lhs -t html

читает грамотный исходный код Haskell, отформатированный с использованием соглашений Markdown, и пишет обычный HTML (без птичьих следов).

pandoc -f markdown+lhs -t html+lhs

записывает HTML с кодом Haskell в птичьих следах, поэтому его можно скопировать и вставить как грамотный исходный код Haskell.

Обратите внимание, что GHC ожидает следы птиц в первом столбце, поэтому грамотные блоки кода с отступом (например, внутри детализированной среды) не будут подхвачены компилятором Haskell.

Pandoc's Markdown

Pandoc понимает расширенную и немного переработанную версию синтаксиса Markdown Джона Грубера . Этот документ объясняет синтаксис, отмечая отличия от стандартного Markdown. Если не указано иное, эти различия можно подавить, используя markdown_strictформат вместо markdown. Расширения можно включить или отключить, чтобы более детально указать поведение. Они описаны ниже. См. Также Расширения выше, чтобы узнать о расширениях, которые также работают с другими форматами.

философия

Markdown разработан таким образом, чтобы его было легко писать и, что еще более важно, легко читать:

Документ в формате Markdown должен быть доступен для публикации как есть, как обычный текст, и не должен выглядеть так, будто он размечен тегами или инструкциями по форматированию. - Джон Грубер

Этот принцип лежит в основе решений pandoc при поиске синтаксиса для таблиц, сносок и других расширений.

Однако есть один аспект, в котором цели pandoc отличаются от первоначальных целей Markdown. В то время как Markdown изначально разрабатывался с учетом генерации HTML, pandoc разработан для нескольких форматов вывода. Таким образом, хотя pandoc позволяет встраивать необработанный HTML, он препятствует этому и предоставляет другие, отличные от HTML способы представления важных элементов документа, таких как списки определений, таблицы, математические данные и сноски.

Пункты

Абзац - это одна или несколько строк текста, за которыми следует одна или несколько пустых строк. Новые строки обрабатываются как пробелы, поэтому вы можете изменять расположение абзацев по своему усмотрению. Если вам нужен жесткий разрыв строки, поставьте два или более пробелов в конце строки.

Расширение: escaped_line_breaks

Обратная косая черта, за которой следует новая строка, также является жестким разрывом строки. Примечание: в ячейках многострочной таблицы и таблицы сетки это единственный способ создать жесткий разрыв строки, поскольку конечные пробелы в ячейках игнорируются.

Заголовки

Есть два типа заголовков: Setext и ATX.

Заголовки в виде сетевого текста

Заголовок в виде сетевого текста - это строка текста, «подчеркнутая» с рядом =знаков (для заголовка первого уровня) или -знаков (для заголовка второго уровня):

A level-one heading
===================

A level-two heading
-------------------

Текст заголовка может содержать встроенное форматирование, например выделение (см. Раздел «Встроенное форматирование» ниже).

Заголовки в стиле ATX

Заголовок в стиле ATX состоит из от одного до шести #знаков и строки текста, за которым может следовать любое количество #знаков. Количество #знаков в начале строки - это уровень заголовка:

## A level-two heading

### A level-three heading ###

Как и в случае заголовков в стиле setext, текст заголовка может содержать форматирование:

# A level-one heading with a [link](/url) and *emphasis*

Расширение: blank_before_header

Стандартный синтаксис Markdown не требует наличия пустой строки перед заголовком. Pandoc действительно этого требует (кроме, конечно, начала документа). Причина этого требования в том, что для a слишком легко #случайно оказаться в начале строки (возможно, из-за переноса строки). Рассмотрим, например:

I like several of their flavors of ice cream:
#22, for example, and #5.

Расширение: space_in_atx_header

Многие реализации Markdown не требуют пробелов между открытием #заголовка ATX и текстом заголовка, так что #5 boltи#hashtag считается заголовком. С этим расширением pandoc действительно требует места.

Расширение: header_attributes

Заголовкам можно присвоить атрибуты с помощью этого синтаксиса в конце строки, содержащей текст заголовка:

{#identifier .class .class key=value key=value}

Таким образом, например, всем следующим заголовкам будет присвоен идентификатор foo:

# My heading {#foo}

## My heading ##    {#foo}

My other heading   {#foo}
---------------

(Этот синтаксис совместим с PHP Markdown Extra .)

Обратите внимание, что хотя этот синтаксис позволяет назначать классы и атрибуты «ключ / значение», авторы обычно не используют всю эту информацию. Идентификаторы, классы и атрибуты "ключ-значение" используются в форматах HTML и HTML, таких как EPUB и slidy. Идентификаторы используются для меток и якорей ссылок в средствах записи LaTeX, ConTeXt, Textile, Jira markup и AsciiDoc.

Заголовки с классом unnumberedне будут нумероваться, даже если --number-sectionsон указан. Одиночный дефис ( -) в контексте атрибута эквивалентен .unnumberedи предпочтительнее в неанглоязычных документах. Так,

# My heading {-}

точно так же, как

# My heading {.unnumbered}

Если unlistedкласс присутствует в дополнение к unnumbered, заголовок не будет включен в оглавление. (В настоящее время эта функция реализована только для определенных форматов: основанных на LaTeX и HTML, PowerPoint и RTF.)

Расширение: implicit_header_references

Pandoc ведет себя так, как если бы ссылочные ссылки были определены для каждого заголовка. Итак, для ссылки на заголовок

# Heading identifiers in HTML

вы можете просто написать

[Heading identifiers in HTML]

или

[Heading identifiers in HTML][]

или

[the section on heading identifiers][heading identifiers in
HTML]

вместо того, чтобы явно указывать идентификатор:

[Heading identifiers in HTML](#heading-identifiers-in-html)

Если имеется несколько заголовков с идентичным текстом, соответствующая ссылка будет ссылаться только на первый, и вам нужно будет использовать явные ссылки для ссылки на другие, как описано выше.

Как и в обычных ссылках, в этих ссылках не учитывается регистр.

# Foo

[foo]: bar

See [foo]

Заблокировать цитаты

Markdown использует соглашения электронной почты для цитирования блоков текста. Цитата из блока - это один или несколько абзацев или других элементов блока (таких как списки или заголовки), где каждой строке предшествует >символ и необязательный пробел. ( >Необязательно начинать с левого поля, но он не должен иметь отступ более трех пробелов.)

> This is a block quote. This
> paragraph has two lines.
>
> 1. This is a list inside a block quote.
> 2. Second item.

>Также разрешена «ленивая» форма, которая требует наличия символа только в первой строке каждого блока:

> This is a block quote. This
paragraph has two lines.

> 1. This is a list inside a block quote.
2. Second item.

Среди элементов блока, которые могут содержаться в цитате блока, есть и другие цитаты блока. То есть блочные кавычки могут быть вложенными:

> This is a block quote.
>
> > A block quote within a block quote.

Если за >символом следует необязательный пробел, этот пробел будет считаться частью маркера цитаты блока, а не частью отступа содержимого. Таким образом, чтобы поместить блок кода с отступом в цитату блока, вам нужно пять пробелов после >:

>     code

Расширение: blank_before_blockquote

Стандартный синтаксис Markdown не требует пустой строки перед цитатой блока. Pandoc действительно этого требует (кроме, конечно, начала документа). Причина этого требования в том, что для a слишком легко >случайно оказаться в начале строки (возможно, из-за переноса строки). Итак, если markdown_strictформат не используется, следующее не создает вложенную цитату блока в pandoc:

> This is a block quote.
>> Nested.

Дословные (кодовые) блоки

Блоки кода с отступом

Блок текста с отступом в четыре пробела (или одна табуляция) рассматривается как дословный текст: то есть специальные символы не вызывают специального форматирования, и все пробелы и разрывы строк сохраняются. Например,

    if (a > 3) {
      moveShip(5 * gravity, DOWN);
    }

Начальный отступ (четыре пробела или одна табуляция) не считается частью дословного текста и удаляется при выводе.

Примечание: пустые строки в дословном тексте не обязательно должны начинаться с четырех пробелов.

Изолированные блоки кода

Расширение: fenced_code_blocks

В дополнении к стандартному отступу кодовых блоков, pandoc поддерживает огорожены блоки кода. Они начинаются строкой из трех или более тильд ( ~) и заканчиваются рядом тильд, длина которых должна быть не меньше длины начальной строки. Все между этими строками рассматривается как код. Никаких отступов не требуется:

~~~~~~~
if (a > 3) {
  moveShip(5 * gravity, DOWN);
}
~~~~~~~

Как и обычные блоки кода, изолированные блоки кода должны отделяться от окружающего текста пустыми строками.

Если сам код содержит ряд тильд или обратных кавычек, просто используйте более длинный ряд тильд или обратных кавычек в начале и в конце:

~~~~~~~~~~~~~~~~
~~~~~~~~~~
code including tildes
~~~~~~~~~~
~~~~~~~~~~~~~~~~

Расширение: backtick_code_blocks

То же, что fenced_code_blocks, но использует обратные кавычки ( `) вместо тильд ( ~).

Расширение: fenced_code_attributes

При желании вы можете прикрепить атрибуты к изолированному блоку кода или блоку кода с обратной кавычкой, используя этот синтаксис:

~~~~ {#mycode .haskell .numberLines startFrom="100"}
qsort []     = []
qsort (x:xs) = qsort (filter (< x) xs) ++ [x] ++
               qsort (filter (>= x) xs)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Вот mycodeэто идентификатор, haskellи numberLinesклассы, и startFromэто атрибут со значением 100. Некоторые форматы вывода могут использовать эту информацию для выделения синтаксиса. В настоящее время единственными выходными форматами, в которых используется эта информация, являются HTML, LaTeX, Docx, Ms и PowerPoint. Если выделение поддерживается для вашего формата вывода и языка, то указанный выше блок кода будет выделен с пронумерованными строками. (Чтобы узнать, какие языки поддерживаются, введите pandoc --list-highlight-languages.) В противном случае приведенный выше блок кода будет выглядеть следующим образом:

<pre id="mycode" class="haskell numberLines" startFrom="100">
  <code>
  ...
  </code>
</pre>

Класс numberLines(или number-lines) приведет к тому, что строки блока кода будут пронумерованы, начиная с 1или значения startFromатрибута. Класс lineAnchors(или line-anchors) приведет к тому, что строки станут интерактивными якорями в выводе HTML.

Для указания языка блока кода также можно использовать форму быстрого доступа:

```haskell
qsort [] = []
```

Это эквивалентно:

``` {.haskell}
qsort [] = []
```

Если fenced_code_attributesрасширение отключено, но входные данные содержат атрибут (ы) класса для блока кода, атрибут первого класса будет напечатан после открывающего ограждения в виде простого слова.

Чтобы предотвратить любое выделение, используйте --no-highlightфлажок. Чтобы установить стиль выделения, используйте --highlight-style. Для получения дополнительной информации о выделении см. Раздел «Подсветка синтаксиса» ниже.

Линейные блоки

Расширение: line_blocks

Блок строки - это последовательность строк, которая начинается с вертикальной черты ( |), за которой следует пробел. Разделение на строки будет сохранено в выводе, как и любые начальные пробелы; в противном случае строки будут отформатированы как Markdown. Это полезно для стихов и адресов:

| The limerick packs laughs anatomical
| In space that is quite economical.
|    But the good ones I've seen
|    So seldom are clean
| And the clean ones so seldom are comical

| 200 Main St.
| Berkeley, CA 94718

При необходимости строки могут быть жестко обернуты, но строка продолжения должна начинаться с пробела.

| The Right Honorable Most Venerable and Righteous Samuel L.
  Constable, Jr.
| 200 Main St.
| Berkeley, CA 94718

Этот синтаксис заимствован из reStructuredText .

Списки

Списки маркеров

Маркированный список - это список элементов маркированного списка. Маркированный элемент списка начинается с пулей ( *, +или -). Вот простой пример:

* one
* two
* three

Это создаст «компактный» список. Если вам нужен «свободный» список, в котором каждый элемент отформатирован как абзац, поместите между элементами пробелы:

* one

* two

* three

Пули не должны быть заподлицо с левым краем; они могут иметь отступ в один, два или три пробела. После маркера должен быть пробел.

Элементы списка выглядят лучше, если последующие строки совпадают с первой строкой (после маркера):

* here is my first
  list item.
* and my second.

Но Markdown также допускает «ленивый» формат:

* here is my first
list item.
* and my second.

Блокировать содержимое в элементах списка

Элемент списка может содержать несколько абзацев и другое содержимое блочного уровня. Однако последующим абзацам должна предшествовать пустая строка с отступом, чтобы соответствовать первому непробельному содержимому после маркера списка.

  * First paragraph.

    Continued.

  * Second paragraph. With a code block, which must be indented
    eight spaces:

        { code }

Исключение: если за маркером списка следует блок кода с отступом, который должен начинаться на 5 пробелов после маркера списка, то последующие абзацы должны начинаться на два столбца после последнего символа маркера списка:

*     code

  continuation paragraph

Элементы списка могут включать другие списки. В этом случае предыдущая пустая строка не обязательна. Вложенный список должен иметь отступ, чтобы соответствовать первому непробельному символу после маркера списка содержащего его элемента.

* fruits
  + apples
    - macintosh
    - red delicious
  + pears
  + peaches
* vegetables
  + broccoli
  + chard

Как отмечалось выше, Markdown позволяет вам «лениво» писать элементы списка вместо того, чтобы делать отступы в строках продолжения. Однако, если в элементе списка несколько абзацев или других блоков, первая строка каждого должна быть с отступом.

+ A lazy, lazy, list
item.

+ Another one; this looks
bad but is legal.

    Second paragraph of second
list item.

Упорядоченные списки

Упорядоченные списки работают так же, как маркированные списки, за исключением того, что элементы начинаются с счетчиков, а не с маркеров.

В стандартном Markdown перечислители представляют собой десятичные числа, за которыми следуют точка и пробел. Сами числа игнорируются, поэтому разницы между этим списком нет:

1.  one
2.  two
3.  three

и этот:

5.  one
7.  two
1.  three

Расширение: fancy_lists

В отличие от стандартного Markdown, pandoc позволяет помечать элементы упорядоченного списка прописными и строчными буквами и римскими цифрами в дополнение к арабским цифрам. Маркеры списка могут быть заключены в круглые скобки или после них может стоять одна правая скобка или точка. Они должны быть отделены от текста, который следует за ними, по крайней мере, одним пробелом, и, если маркер списка представляет собой заглавную букву с точкой, по крайней мере, двумя пробелами. 1

fancy_listsРасширение также позволяет « #» , которые будут использоваться в качестве упорядоченного списка маркера вместо цифры:

#. one
#. two

Расширение: startnum

Pandoc также обращает внимание на тип используемого маркера списка и начальный номер, и оба они сохраняются, где это возможно, в выходном формате. Таким образом, следующее дает список с числами, за которыми следует одинарная скобка, начиная с 9, и подсписок с римскими цифрами в нижнем регистре:

 9)  Ninth
10)  Tenth
11)  Eleventh
       i. subone
      ii. subtwo
     iii. subthree

Pandoc будет начинать новый список каждый раз, когда используется другой тип маркера списка. Итак, следующее создаст три списка:

(2) Two
(5) Three
1.  Four
*   Five

Если требуются маркеры списка по умолчанию, используйте #.:

#.  one
#.  two
#.  three

Расширение: task_lists

Pandoc поддерживает списки задач, используя синтаксис GitHub-Flavored Markdown.

- [ ] an unchecked task list item
- [x] checked item

Списки определений

Расширение: definition_lists

Pandoc поддерживает списки определений, используя синтаксис PHP Markdown Extra с некоторыми расширениями. 2

Term 1

:   Definition 1

Term 2 with *inline markup*

:   Definition 2

        { some code, part of Definition 2 }

    Third paragraph of definition 2.

Каждый термин должен умещаться в одной строке, за которой при желании может следовать пустая строка, а за ней должно следовать одно или несколько определений. Определение начинается с двоеточия или тильды, которые могут иметь отступ в один или два пробела.

Термин может иметь несколько определений, и каждое определение может состоять из одного или нескольких элементов блока (абзац, блок кода, список и т. Д.), Каждый из которых имеет четыре пробела с отступом или одну позицию табуляции. Тело определения (включая первую строку, кроме двоеточия или тильды) должно иметь отступ в четыре пробела. Однако, как и в случае с другими списками Markdown, вы можете «лениво» опускать отступы, за исключением начала абзаца или другого элемента блока:

Term 1

:   Definition
with lazy continuation.

    Second paragraph of the definition.

Если вы оставите пробел перед определением (как в примере выше), текст определения будет рассматриваться как абзац. В некоторых форматах вывода это будет означать больший интервал между парами термин / определение. Чтобы получить более компактный список определений, опустите пробел перед определением:

Term 1
  ~ Definition 1

Term 2
  ~ Definition 2a
  ~ Definition 2b

Обратите внимание, что требуется пробел между элементами в списке определений. (Вариант, который ослабляет это требование, но запрещает «ленивую» жесткую упаковку, может быть активирован с помощью compact_definition_lists: см. Расширения , отличные от pandoc , ниже.)

Нумерованные примеры списков

Расширение: example_lists

Специальный маркер списка @может использоваться для последовательно пронумерованных примеров. Первый элемент списка с @маркером будет иметь номер «1», следующий «2» и т. Д. По всему документу. Пронумерованные примеры не обязательно должны быть в едином списке; каждый новый список @будет использоваться там, где останавливался последний. Так, например:

(@)  My first example will be numbered (1).
(@)  My second example will be numbered (2).

Explanation of examples.

(@)  My third example will be numbered (3).

Пронумерованные примеры могут быть помечены и ссылаться на них в другом месте документа:

(@good)  This is a good example.

As (@good) illustrates, ...

Метка может быть любой строкой, состоящей из буквенно-цифровых символов, подчеркиваний или дефисов.

Примечание: продолжение абзацев в примерах списков всегда должно иметь отступ в четыре пробела, независимо от длины маркера списка. То есть списки примеров всегда ведут себя так, как если бы four_space_ruleрасширение было установлено. Это связано с тем, что метки в примерах имеют тенденцию быть длинными, и делать отступ до первого непробельного символа после метки было бы неудобно.

Компактные и свободные списки

Pandoc ведет себя иначе, чем Markdown.plв некоторых «крайних случаях», связанных со списками. Рассмотрим этот источник:

+   First
+   Second:
    -   Fee
    -   Fie
    -   Foe

+   Third

Pandoc преобразует это в «компактный список» (без <p>тегов вокруг «Первый», «Второй» или «Третий»), в то время как Markdown помещает <p>теги вокруг «Второго» и «Третьего» (но не «Первого») из-за пустое пространство вокруг «Третьего». Pandoc следует простому правилу: если за текстом следует пустая строка, он рассматривается как абзац. Поскольку за «Вторым» следует список, а не пустая строка, он не рассматривается как абзац. Тот факт, что после списка идет пустая строка, не имеет значения. (Примечание: Pandoc работает таким образом, даже если markdown_strictформат указан. Такое поведение согласуется с официальным описанием синтаксиса Markdown, даже если оно отличается от Markdown.pl.)

Завершение списка

Что, если вы хотите поместить блок кода с отступом после списка?

-   item one
-   item two

    { my code block }

Беда! Здесь pandoc (как и другие реализации Markdown) будет рассматриваться { my code block }как второй абзац пункта два, а не как блок кода.

Чтобы «отрезать» список после пункта два, вы можете вставить некоторый контент без отступов, например HTML-комментарий, который не будет производить видимый вывод в любом формате:

-   item one
-   item two

<!-- end of list -->

    { my code block }

Вы можете использовать тот же прием, если хотите два последовательных списка вместо одного большого:

1.  one
2.  two
3.  three

<!-- -->

1.  uno
2.  dos
3.  tres

таблицы

Можно использовать четыре вида таблиц. Первые три вида предполагают использование шрифта фиксированной ширины, например Courier. Четвертый тип можно использовать со шрифтами с пропорциональным интервалом, поскольку он не требует выравнивания столбцов.

Расширение: table_captions

По желанию, заголовок может быть снабжен всеми 4 видами таблиц (как показано в примерах ниже). Заголовок - это абзац, начинающийся со строки Table:(или просто :), который будет удален. Он может появиться как до, так и после таблицы.

Расширение: simple_tables

Простые таблицы выглядят так:

  Right     Left     Center     Default
-------     ------ ----------   -------
     12     12        12            12
    123     123       123          123
      1     1          1             1

Table:  Demonstration of simple table syntax.

Каждая строка заголовка и таблицы должна умещаться в одной строке. Выравнивание столбцов определяется положением текста заголовка относительно пунктирной линии под ним: 3

  • Если пунктирная линия находится на одном уровне с текстом заголовка справа, но выходит за ее пределы слева, столбец выравнивается по правому краю.
  • Если пунктирная линия находится на одном уровне с текстом заголовка с левой стороны, но выходит за его пределы справа, столбец выравнивается по левому краю.
  • Если пунктирная линия выходит за пределы текста заголовка с обеих сторон, столбец центрируется.
  • Если пунктирная линия находится на одном уровне с текстом заголовка с обеих сторон, используется выравнивание по умолчанию (в большинстве случаев это будет оставлено).

Таблица должна заканчиваться пустой строкой или строкой тире, за которой следует пустая строка.

Строку заголовка столбца можно не указывать, если в конце таблицы используется пунктирная линия. Например:

-------     ------ ----------   -------
     12     12        12             12
    123     123       123           123
      1     1          1              1
-------     ------ ----------   -------

Если строка заголовка опущена, выравнивание столбцов определяется на основе первой строки тела таблицы. Итак, в приведенных выше таблицах столбцы будут выровнены по правому, левому, центральному и правому краям соответственно.

Расширение: multiline_tables

Многострочные таблицы позволяют строкам заголовка и таблицы занимать несколько строк текста (но ячейки, охватывающие несколько столбцов или строк таблицы, не поддерживаются). Вот пример:

-------------------------------------------------------------
 Centered   Default           Right Left
  Header    Aligned         Aligned Aligned
----------- ------- --------------- -------------------------
   First    row                12.0 Example of a row that
                                    spans multiple lines.

  Second    row                 5.0 Here's another one. Note
                                    the blank line between
                                    rows.
-------------------------------------------------------------

Table: Here's the caption. It, too, may span
multiple lines.

Они работают как простые таблицы, но со следующими отличиями:

  • Они должны начинаться с ряда дефисов перед текстом заголовка (если строка заголовка не указана).
  • Они должны заканчиваться строкой тире, а затем пустой строкой.
  • Строки должны быть разделены пустыми строками.

В многострочных таблицах анализатор таблиц обращает внимание на ширину столбцов, и авторы пытаются воспроизвести эту относительную ширину в выводе. Итак, если вы обнаружите, что один из столбцов в выводе слишком узкий, попробуйте расширить его в источнике Markdown.

Заголовок может быть опущен как в многострочных, так и в простых таблицах:

----------- ------- --------------- -------------------------
   First    row                12.0 Example of a row that
                                    spans multiple lines.

  Second    row                 5.0 Here's another one. Note
                                    the blank line between
                                    rows.
----------- ------- --------------- -------------------------

: Here's a multiline table without a header.

В многострочной таблице может быть только одна строка, но после нее должна следовать пустая строка (а затем строка дефисов, завершающая таблицу), или же таблица может быть интерпретирована как простая таблица.

| Bananas       | $1.34         | built-in wrapper   |
+---------------+---------------+--------------------+

Для таблиц без заголовков двоеточие ставится вместо этого в верхней строке:

+--------------:+:--------------+:------------------:+
| Right         | Left          | Centered           |
+---------------+---------------+--------------------+
Ограничения таблицы сетки

Pandoc не поддерживает сеточные таблицы с диапазоном строк или столбцов. Это означает, что Pandoc не поддерживает ни переменное количество столбцов в строках, ни переменное количество строк в столбцах. Все таблицы сетки должны иметь одинаковое количество столбцов в каждой строке и одинаковое количество строк в каждом столбце. Например, образцы таблиц сетки Docutils не будут отображаться должным образом с помощью Pandoc.

Расширение: pipe_tables

Таблицы труб выглядят так:

| Right | Left | Default | Center |
|------:|:-----|---------|:------:|
|   12  |  12  |    12   |    12  |
|  123  |  123 |   123   |   123  |
|    1  |    1 |     1   |     1  |

  : Demonstration of pipe table syntax.

Синтаксис идентичен таблицам PHP Markdown Extra . Символы начала и конца вертикальной черты необязательны, но между всеми столбцами должны быть вертикальные черты. Двоеточие обозначают выравнивание столбца, как показано. Заголовок не может быть пропущен. Чтобы имитировать таблицу без заголовка, включите заголовок с пустыми ячейками.

Поскольку трубы указывают границы столбцов, столбцы не нужно выравнивать по вертикали, как в приведенном выше примере. Итак, это совершенно законная (хотя и уродливая) таблица каналов:

fruit| price
-----|-----:
apple|2.05
pear|1.37
orange|3.09

Ячейки таблиц каналов не могут содержать блочные элементы, такие как абзацы и списки, и не могут занимать несколько строк. Если вертикальная таблица содержит строку, содержимое Markdown которой шире ширины столбца (см. --columns), То таблица займет всю ширину текста, а содержимое ячейки будет перенесено, а относительная ширина ячеек будет определяться количеством тире в линия, отделяющая заголовок таблицы от тела таблицы. (Например, ---|-можно сделать первый столбец 3/4 и второй столбец 1/4 от полной ширины текста.) С другой стороны, если ни одна строка не шире ширины столбца, то содержимое ячейки не будет перенесено, а ячейки размер будет соответствовать их содержимому.

Примечание: pandoc также распознает таблицы каналов следующей формы, которые могут быть созданы Emacs orgtbl-mode:

| One | Two   |
|-----+-------|
| my  | table |
| is  | nice  |

Разница в том, что +используется вместо |. Другие функции orgtbl не поддерживаются. В частности, чтобы получить выравнивание столбцов не по умолчанию, вам нужно добавить двоеточия, как указано выше.

Блоки метаданных

Расширение: pandoc_title_block

Если файл начинается с основной надписи

% title
% author(s) (separated by semicolons)
% date

он будет анализироваться как библиографическая информация, а не как обычный текст. (Он будет использоваться, например, в заголовке автономного вывода LaTeX или HTML.) Блок может содержать только заголовок, заголовок и автора или все три элемента. Если вы хотите указать автора, но без заголовка, или заголовок и дату, но без автора, вам понадобится пустая строка:

%
% Author
% My title
%
% June 15, 2006

Заголовок может занимать несколько строк, но строки продолжения должны начинаться с ведущего пробела, таким образом:

% My title
  on multiple lines

Если у документа есть несколько авторов, авторы могут быть помещены в отдельные строки с начальным пробелом, разделены точкой с запятой или и тем, и другим. Итак, все следующее эквивалентно:

% Author One
  Author Two
% Author One; Author Two
% Author One;
  Author Two

Дата должна умещаться в одной строке.

Все три поля метаданных могут содержать стандартное встроенное форматирование (курсив, ссылки, сноски и т. Д.).

Блоки заголовков всегда будут анализироваться, но они повлияют на вывод, только если выбрана опция --standalone( -s). При выводе HTML заголовки будут появляться дважды: один раз в заголовке документа - это заголовок, который будет отображаться в верхней части окна в браузере, и один раз в начале тела документа. Заголовок в заголовке документа может иметь дополнительный префикс ( --title-prefixили -Tвариант). Заголовок в теле отображается как элемент H1 с классом «title», поэтому его можно подавить или переформатировать с помощью CSS. Если префикс заголовка указан с помощью -Tи в документе не отображается блок заголовка, префикс заголовка будет использоваться сам по себе как заголовок HTML.

Автор страницы руководства извлекает заголовок, номер раздела страницы руководства и другую информацию верхнего и нижнего колонтитула из строки заголовка. Предполагается, что заголовок является первым словом в строке заголовка, которое может дополнительно заканчиваться (однозначным) номером раздела в круглых скобках. (Между заголовком и круглыми скобками не должно быть пробелов.) Все, что находится после этого, считается дополнительным текстом нижнего и верхнего колонтитула. Для |отделения текста нижнего колонтитула от текста верхнего колонтитула следует использовать одинарную вертикальную черту ( ). Таким образом,

% PANDOC(1)

откроет страницу руководства с заголовком PANDOCи разделом 1.

% PANDOC(1) Pandoc User Manuals

в нижнем колонтитуле также будет «Руководства пользователя Pandoc».

% PANDOC(1) Pandoc User Manuals | Version 4.0

в шапке также будет «Версия 4.0».

Расширение: yaml_metadata_block

Блок метаданных YAML - это действительный объект YAML, ограниченный линией из трех дефисов ( ---) вверху и линией из трех дефисов ( ---) или трех точек ( ...) внизу. Блок метаданных YAML может находиться в любом месте документа, но если он находится не в начале, ему должна предшествовать пустая строка. (Обратите внимание, что из-за того, как pandoc объединяет входные файлы, когда их несколько, вы также можете сохранить метаданные в отдельном файле YAML и передать их pandoc в качестве аргумента вместе с вашими файлами Markdown:

pandoc chap1.md chap2.md chap3.md metadata.yaml -s -o book.html

Просто убедитесь, что файл YAML начинается ---и заканчивается на ---или .... В качестве альтернативы вы можете использовать эту --metadata-fileопцию. Однако, используя этот подход, вы не можете ссылаться на контент (например, сноски) из основного входного документа уценки.

Метаданные будут взяты из полей объекта YAML и добавлены к любым существующим метаданным документа. Метаданные могут содержать списки и объекты (вложенные произвольно), но все строковые скаляры будут интерпретироваться как Markdown. Поля, имена которых заканчиваются знаком подчеркивания, будут игнорироваться pandoc. (Они могут быть отведена роль внешними процессорами.) Имена полей не должны быть интерпретированы как числа YAML или логических значений (так, например, yes, True, и 15не могут быть использованы в качестве имен полей).

Документ может содержать несколько блоков метаданных. Если два блока метаданных попытаются установить одно и то же поле, будет взято значение из второго блока.

Когда pandoc используется с -t markdownдля создания документа Markdown, блок метаданных YAML будет создан только в том случае, если используется этот -s/--standaloneпараметр. Все метаданные появятся в одном блоке в начале документа.

Обратите внимание, что необходимо соблюдать правила экранирования YAML . Так, например, если заголовок содержит двоеточие, его необходимо заключить в кавычки. Символ вертикальной черты ( |) может использоваться для начала блока с отступом, который будет интерпретироваться буквально, без необходимости экранирования. Эта форма необходима, если поле содержит пустые строки или форматирование на уровне блока:

---
title:  'This is the title: it contains a colon'
author:
- Author One
- Author Two
keywords: [nothing, nothingness]
abstract: |
  This is the abstract.

  It consists of two paragraphs.
...

Переменные шаблона будут установлены автоматически из метаданных. Так, например, при написании HTML переменная abstractбудет установлена ​​в HTML-эквивалент Markdown в abstractполе:

<p>This is the abstract.</p>
<p>It consists of two paragraphs.</p>

Переменные могут содержать произвольные структуры YAML, но шаблон должен соответствовать этой структуре. authorПеременной в шаблонах по умолчанию ожидает , что список простых или строку, но может быть изменен для поддержки более сложных структур. Следующая комбинация, например, добавит принадлежность к автору, если она указана:

---
title: The document title
author:
- name: Author One
  affiliation: University of Somewhere
- name: Author Two
  affiliation: University of Nowhere
...

Чтобы использовать структурированных авторов в приведенном выше примере, вам понадобится настраиваемый шаблон:

$for(author)$
$if(author.name)$
$author.name$$if(author.affiliation)$ ($author.affiliation$)$endif$
$else$
$author$
$endif$
$endfor$

Необработанный контент для включения в заголовок документа можно указать с помощью header-includes; однако важно пометить это содержимое как необработанный код для определенного формата вывода, используя raw_attributeрасширение ), иначе он будет интерпретирован как уценка. Например:

header-includes:
- |
  ```{=latex}
  \let\oldsection\section
  \renewcommand{\section}[1]{\clearpage\oldsection{#1}}
  ```

Обратная косая черта экранирует

Расширение: all_symbols_escapable

За исключением кода внутри блока кода или встроенного кода, любой знак пунктуации или пробел, которому предшествует обратная косая черта, будет обрабатываться буквально, даже если он обычно указывает на форматирование. Так, например, если написать

*\*hello\**

один получит

<em>*hello*</em>

вместо того

<strong>hello</strong>

Это правило легче запомнить, чем стандартное правило Markdown, которое позволяет экранировать обратную косую черту только для следующих символов:

\`*_{}[]()>#+-.!

(Однако, если используется markdown_strictформат, будет использоваться стандартное правило Markdown.)

Пробел с обратной косой чертой анализируется как неразрывный. В выводе TeX он будет выглядеть как ~. В выводе HTML и XML он будет отображаться как буквальный неразрывный пробел в кодировке Unicode (обратите внимание, что он будет фактически выглядеть «невидимым» в сгенерированном источнике HTML; вы все равно можете использовать параметр --asciiкомандной строки, чтобы он отображался как явный объект ).

Новая строка с экранированной обратной косой чертой (т.е. обратная косая черта, встречающаяся в конце строки) анализируется как жесткий разрыв строки. Он будет отображаться в выводе TeX как \\и в HTML как <br />. Это хорошая альтернатива «невидимому» способу Markdown указывать жесткие разрывы строк с помощью двух конечных пробелов в строке.

Экраны с обратной косой чертой не работают в дословном контексте.

Встроенное форматирование

акцент

Чтобы выделить текст, заключите его в *s или _, например:

This text is _emphasized with underscores_, and this
is *emphasized with asterisks*.

Двойной *или _производит сильный акцент :

This is **strong emphasis** and __with underscores__.

Символ *или, _окруженный пробелами или экранированный обратной косой чертой, не вызовет выделения:

This is * not emphasized *, and \*neither is this\*.

Расширение: intraword_underscores

Поскольку _иногда используется внутри слов и идентификаторов, pandoc не интерпретирует, _окруженный буквенно-цифровыми символами, как маркер выделения. Если вы хотите выделить часть слова, используйте *:

feas*ible*, not feas*able*.

аут

Расширение: strikeout

Чтобы вычеркнуть фрагмент текста горизонтальной линией, начните и завершите его знаком ~~. Так, например,

This ~~is deleted text.~~

Верхние и нижние индексы

Расширение: superscript,subscript

Надстрочные символы могут быть написаны путем окружения надстрочного текста ^символами; Подстрочные индексы могут быть написаны путем окружения подстрочного текста ~символами. Так, например,

H~2~O is a liquid.  2^10^ is 1024.

Текст между ними ^...^или ~...~не может содержать пробелов или символов новой строки. Если текст с надстрочным или подстрочным индексом содержит пробелы, эти пробелы должны быть экранированы обратной косой чертой. (Это сделано для предотвращения случайного надстрочного и подстрочного индексов из-за обычного использования ~и ^, а также плохого взаимодействия со сносками.) Таким образом, если вы хотите, чтобы в подстрочных индексах была буква P с «кошкой», используйте P~a\ cat~, not P~a cat~.

дословный

Чтобы сделать короткий текст дословным, поместите его в обратные кавычки:

What is the difference between `>>=` and `>>`?

Если дословный текст включает обратную кавычку, используйте двойные обратные кавычки:

Here is a literal backtick `` ` ``.

(Пробелы после открывающих обратных кавычек и перед закрывающими обратными кавычками будут проигнорированы.)

Общее правило состоит в том, что дословный интервал начинается со строки последовательных обратных кавычек (необязательно с последующим пробелом) и заканчивается строкой с таким же количеством обратных кавычек (необязательно с пробелом).

Обратите внимание, что обратная косая черта (и другие конструкции Markdown) не работают в буквальном контексте:

This is a backslash followed by an asterisk: `\*`.

Расширение: inline_code_attributes

Атрибуты могут быть прикреплены к дословному тексту, как и к изолированным блокам кода :

`<$>`{.haskell}

Маленькие шапки

Чтобы писать заглавные буквы, используйте smallcapsкласс:

[Small caps]{.smallcaps}

Или без bracketed_spansрасширения:

<span class="smallcaps">Small caps</span>

Для совместимости с другими разновидностями Markdown также поддерживается CSS:

<span style="font-variant:small-caps;">Small caps</span>

Это будет работать во всех выходных форматах, поддерживающих маленькие заглавные буквы.

математический

Расширение: tex_math_dollars

Все, что находится между двумя $символами, будет рассматриваться как математика TeX. Открытие $должно иметь непробельный символ непосредственно справа от него, в то время как закрытие $должно иметь непробельный символ непосредственно слева от него и не должно сразу сопровождаться цифрой. Таким образом, $20,000 and $30,000не будет анализироваться как математика. Если по какой-то причине вам нужно заключить текст в буквальные $символы, экранируйте их с помощью обратной косой черты, и они не будут рассматриваться как математические разделители.

Для математического отображения используйте $$разделители. (В этом случае разделители могут быть отделены от формулы пробелом.)

Математика TeX будет напечатана во всех выходных форматах. Как он отображается, зависит от формата вывода:

Латекс
Он будет дословно заключен в \(...\)(для встроенных математических вычислений) или \[...\](для отображаемых математических вычислений).
Markdown, режим Emacs Org, ConTeXt, ZimWiki
Он будет дословно заключен в $...$(для встроенных математических вычислений) или $$...$$(для отображаемых математических вычислений).
XWiki
Он появится дословно в окружении {{formula}}..{{/formula}}.
ReStructuredText
Он будет отображаться с использованием интерпретируемой текстовой роли:math: .
AsciiDoc
Для формата вывода AsciiDoc ( -t asciidoc) он будет дословно заключен в latexmath:[$...$](для встроенных математических вычислений) или [latexmath]++++\[...\]+++(для отображения математических вычислений). Для формата вывода AsciiDoctor ( -t asciidoctor) разделители LaTex ( $..$и \[..\]) опускаются.
Texinfo
Он будет отображаться внутри @mathкоманды.
roff man, разметка Jira
Он будет воспроизводиться дословно без $'s.
MediaWiki, «ДокуВики»
Он будет отображаться внутри <math>тегов.
текстильный
Он будет отображаться внутри <span class="math">тегов.
RTF, OpenDocument
Если возможно, он будет отображаться с использованием символов Юникода, а в противном случае будет отображаться дословно.
ODT
По возможности он будет отображаться с использованием MathML.
DocBook
Если используется --mathmlфлаг, он будет отображаться с использованием MathML в теге inlineequationили informalequation. В противном случае он будет отображаться, если это возможно, с использованием символов Юникода.
Docx
Он будет отображаться с использованием математической разметки OMML.
FictionBook2
Если этот --webtexпараметр используется, формулы отображаются в виде изображений с помощью CodeCogs или другой совместимой веб-службы, загружаются и встраиваются в электронную книгу. В противном случае они будут отображаться дословно.
HTML, Slidy, DZSlides, S5, EPUB
Способ отображения математических операций в HTML будет зависеть от выбранных параметров командной строки. Поэтому см. Математический рендеринг в HTML выше.

Необработанный HTML

Расширение: raw_html

Markdown позволяет вставлять необработанный HTML (или DocBook) в любом месте документа ( за исключением стенографических контекстов, где <, >и &интерпретируются в буквальном смысле). (Технически это не расширение, поскольку стандартный Markdown позволяет это, но он был сделан как расширение, так что при желании его можно отключить.)

Необработанный HTML передается без изменений в HTML, S5, Slidy, Slideous, DZSlides, EPUB, Markdown, CommonMark, режиме Emacs Org и текстовом выводе и подавляется в других форматах.

Более подробный способ включения необработанного HTML в документ Markdown см. В raw_attributeрасширении .

В формате CommonMark, если raw_htmlон включен, верхние индексы, нижние индексы, зачеркивания и заглавные буквы будут представлены как HTML. В противном случае будут использоваться резервные копии в виде обычного текста. Обратите внимание, что даже если raw_htmlон отключен, таблицы будут отображаться с синтаксисом HTML, если они не могут использовать синтаксис конвейерной передачи.

Расширение: markdown_in_html_blocks

Стандартный Markdown позволяет включать «блоки» HTML: блоки HTML между сбалансированными тегами, которые отделены от окружающего текста пустыми строками и начинаются и заканчиваются на левом поле. Внутри этих блоков все интерпретируется как HTML, а не как Markdown; так (например), *не означает акцента.

Pandoc ведет себя так, когда используется markdown_strictформат; но по умолчанию pandoc интерпретирует материал между тегами блока HTML как Markdown. Так, например, pandoc превратится

<table>
<tr>
<td>*one*</td>
<td>[a link](https://google.com)</td>
</tr>
</table>

в

<table>
<tr>
<td><em>one</em></td>
<td><a href="https://google.com">a link</a></td>
</tr>
</table>

тогда как Markdown.plсохранит его как есть.

Существует одно исключение из этого правила: текст между <script>и <style>теги не интерпретируется как Markdown.

Этот отход от стандартного Markdown должен упростить смешивание Markdown с блочными элементами HTML. Например, можно окружить блок текста Markdown <div>тегами, не препятствуя его интерпретации как Markdown.

Расширение: native_divs

Используйте собственные Divблоки pandoc для содержимого внутри <div>тегов. По большей части это должно давать тот же результат, что и markdown_in_html_blocks, но упрощает написание фильтров pandoc для управления группами блоков.

Расширение: native_spans

Используйте собственные Spanблоки pandoc для содержимого внутри <span>тегов. По большей части это должно дать тот же результат, что и raw_html, но упрощает написание фильтров pandoc для управления группами встроенных строк.

Расширение: raw_tex

В дополнение к необработанному HTML, pandoc позволяет включать в документ необработанные LaTeX, TeX и ConTeXt. Встроенные команды TeX будут сохранены и переданы авторам LaTeX и ConTeXt без изменений. Так, например, вы можете использовать LaTeX для включения цитат BibTeX:

This result was proved in \cite{jones.1967}.

Обратите внимание, что в среде LaTeX, например

\begin{tabular}{|l|l|}\hline
Age & Frequency \\ \hline
18--25  & 15 \\
26--35  & 33 \\
36--45  & 22 \\ \hline
\end{tabular}

материал между начальным и конечным тегами будет интерпретироваться как необработанный LaTeX, а не как Markdown.

Для более явного и гибкого способа включения необработанного TeX в документ Markdown см. raw_attributeРасширение .

Встроенный LaTeX игнорируется в выходных форматах, отличных от Markdown, LaTeX, Emacs Org mode и ConTeXt.

Общий необработанный атрибут

Расширение: raw_attribute

Встроенные промежутки и изолированные блоки кода с особым типом атрибута будут анализироваться как необработанный контент в указанном формате. Например, следующее создает необработанный msблок roff :

```{=ms}
.MYMACRO
blah blah
```

И следующее создает необработанный htmlвстроенный элемент:

This is `<a>html</a>`{=html}

Это может быть полезно для вставки необработанного xml в docxдокументы, например, разрыв страницы:

```{=openxml}
<w:p>
  <w:r>
    <w:br w:type="page"/>
  </w:r>
</w:p>
```

Имя формата должно соответствовать имени целевого формата ( -t/--toсписок или использование см. Выше pandoc --list-output-formats). Используйте openxmlдля docxвывода, opendocumentдля odtпроизводства, html5для epub3производства, html4для epub2производства, и latex, beamer, ms, или html5для pdfвывода ( в зависимости от того, что вы используете для --pdf-engine).

Это расширение предполагает, что соответствующий тип встроенного кода или изолированного блока кода включен. Таким образом, например, для использования необработанного атрибута с блоком кода обратной кавычки backtick_code_blocksнеобходимо включить.

Необработанный атрибут нельзя комбинировать с обычными атрибутами.

Макросы LaTeX

Расширение: latex_macros

Когда это расширение включено, pandoc будет анализировать определения макросов LaTeX и применять полученные макросы ко всей математике LaTeX и необработанному LaTeX. Так, например, следующее будет работать во всех форматах вывода, а не только в LaTeX:

\newcommand{\tuple}[1]{\langle #1 \rangle}

$\tuple{a, b, c}$

Обратите внимание, что макросы LaTeX не будут применяться, если они встречаются внутри необработанного диапазона или блока, отмеченного raw_attributeрасширением .

Когда latex_macrosон отключен, необработанный LaTeX и математика не будут иметь макросов. Обычно это лучший подход, когда вы ориентируетесь на LaTeX или PDF.

Определения макросов в LaTeX будут передаваться как необработанный LaTeX, только если latex_macrosон не включен. Определения макросов в исходном коде Markdown (или в других разрешенных форматах raw_tex) будут передаваться независимо от того latex_macros, включен ли параметр .

связи

Markdown позволяет указывать ссылки несколькими способами.

Автоматические ссылки

Если вы заключите URL или адрес электронной почты в острые скобки, он станет ссылкой:

<https://google.com>
<sam@green.eggs.ham>

Встроенные ссылки

Встроенная ссылка состоит из текста ссылки в квадратных скобках, за которым следует URL-адрес в круглых скобках. (Необязательно, за URL-адресом может следовать заголовок ссылки в кавычках.)

This is an [inline link](/url), and here's [one with
a title](https://fsf.org "click here for a good time!").

Между частью в скобках и частью в скобках не должно быть места. Текст ссылки может содержать форматирование (например, выделение), но заголовок не может.

Адреса электронной почты во встроенных ссылках не определяются автоматически, поэтому они должны иметь префикс mailto:

[Write me!](mailto:sam@green.eggs.ham)

Справочные ссылки

Явная ссылка звено имеет две части, саму ссылку и определение линии связи, который может произойти в другом месте в документе (либо до , либо после того, как по ссылке).

Ссылка состоит из текста ссылки в квадратных скобках, за которым следует метка в квадратных скобках. (Между ними не может быть пробела, если spaced_reference_linksне включено расширение.) Определение ссылки состоит из заключенной в квадратные скобки метки, за которой следуют двоеточие и пробел, за которыми следует URL-адрес, и, возможно, (после пробела) заголовок ссылки в кавычках. или в скобках. Метка не должна анализироваться как цитата (при условии, что citationsрасширение включено): цитаты имеют приоритет над метками ссылок.

Вот некоторые примеры:

[my label 1]: /foo/bar.html  "My title, optional"
[my label 2]: /foo
[my label 3]: https://fsf.org (The free software foundation)
[my label 4]: /bar#special  'A title in single quotes'

URL-адрес может быть дополнительно заключен в угловые скобки:

[my label 5]: <http://foo.bar.baz>

Заголовок может идти в следующей строке:

[my label 3]: https://fsf.org
  "The free software foundation"

Обратите внимание, что метки ссылок не чувствительны к регистру. Итак, это будет работать:

Here is [my link][FOO]

[Foo]: /bar/baz

В неявной ссылочной ссылке вторая пара скобок пуста:

See [my website][].

[my website]: http://foo.bar.baz

Примечание. В Markdown.plбольшинстве других реализаций Markdown определения ссылочных ссылок не могут встречаться во вложенных конструкциях, таких как элементы списка или блочные кавычки. Пандок снимает это произвольное кажущееся ограничение. Таким образом, в pandoc нормально, но не в большинстве других реализаций:

> My block [quote].
>
> [quote]: /foo

Расширение: shortcut_reference_links

В ссылке на ярлык вторую пару скобок можно полностью опустить:

See [my website].

[my website]: http://foo.bar.baz

Внутренние ссылки

Для ссылки на другой раздел того же документа используйте автоматически сгенерированный идентификатор (см. Идентификаторы заголовков ). Например:

See the [Introduction](#introduction).

или

See the [Introduction].

[Introduction]: #introduction

В настоящее время внутренние ссылки поддерживаются для форматов HTML (включая слайд-шоу HTML и EPUB), LaTeX и ConTeXt.

Изображений

Ссылка, перед которой стоит !символ, будет рассматриваться как изображение. Текст ссылки будет использоваться в качестве альтернативного текста изображения:

![la lune](lalune.jpg "Voyage to the moon")

![movie reel]

[movie reel]: movie.gif

Расширение: implicit_figures

Изображение с непустым замещающим текстом, встречающееся само по себе в абзаце, будет отображаться как рисунок с подписью. Альтернативный текст изображения будет использоваться в качестве заголовка.

![This is the caption](/url/of/image.png)

Как это отображается, зависит от формата вывода. Некоторые форматы вывода (например, RTF) еще не поддерживают цифры. В этих форматах вы просто получите изображение в абзаце, без подписи.

Если вам просто нужно обычное встроенное изображение, просто убедитесь, что это не единственное, что есть в абзаце. Один из способов сделать это - вставить неразрывный пробел после изображения:

![This image won't be a figure](/url/of/image.png)\

Обратите внимание, что в слайд-шоу Show.js изображение в абзаце, которое имеет stretchкласс, будет заполнять экран, а теги caption и figure будут опущены.

Расширение: link_attributes

Атрибуты могут быть установлены для ссылок и изображений:

An inline ![image](foo.jpg){#id .class width=30 height=20px}
and a reference ![image][ref] with attributes.

[ref]: foo.jpg "optional title" {#id .class key=val key2="val 2"}

(Этот синтаксис совместим с PHP Markdown Extra , когда только #idи .classиспользуются) .

Для HTML и EPUB все известные атрибуты HTML5, кроме widthи height(но включая srcsetи sizes), передаются как есть. Неизвестные атрибуты передаются как настраиваемые атрибуты с data-добавлением в начале. Остальные средства записи игнорируют атрибуты, которые специально не поддерживаются их форматом вывода.

Эти widthи heightатрибуты на изображениях, специально обрабатываются. При использовании без единицы измерения предполагается, что единицей измерения являются пиксели. Однако любой из следующих идентификаторов единиц могут быть использованы: px, cm, mm, in, inchи %. Между числом и единицей не должно быть пробелов. Например:

![](file.jpg){ width=50% }
  • Размеры могут быть преобразованы в форму, совместимую с выходным форматом (например, размеры, указанные в пикселях, будут преобразованы в дюймы при преобразовании HTML в LaTeX). Этот --dpiпараметр влияет на преобразование пикселей в физические измерения (по умолчанию предполагается 96 точек на дюйм, если само изображение не содержит информацию о точках на дюйм).
  • %Блок , как правило , относительно некоторого свободного пространства. Например, приведенный выше пример будет отображаться следующим образом.
    • HTML: <img href="file.jpg" style="width: 50%;" />
    • LaTeX: \includegraphics[width=0.5\textwidth,height=\textheight]{file.jpg}(Если вы используете настраиваемый шаблон, вам нужно настроить graphicxкак в шаблоне по умолчанию.)
    • ConTeXt: \externalfigure[file.jpg][width=0.5\textwidth]
  • Некоторые выходные форматы имеют понятие класса ( ConTeXt ), уникального идентификатора (LaTeX \caption) или того и другого (HTML).
  • Если атрибуты widthили heightне указаны, следует обратиться к разрешению изображения и метаданным dpi, встроенным в файл изображения.

Дивы и пролеты

Использование native_divsи native_spansрасширение (см выше ), HTML синтаксис может быть использован как часть уценки создавать собственные Divи Spanэлементы в pandoc AST (в отличие от сырого HTML). Однако есть и более удобный синтаксис:

Расширение: fenced_divs

Разрешить специальный изолированный синтаксис для собственных Divблоков. Div начинается с забора, содержащего как минимум три последовательных двоеточия плюс некоторые атрибуты. Атрибуты могут дополнительно сопровождаться другой цепочкой последовательных двоеточий. Синтаксис атрибута такой же, как в изолированных блоках кода (см. Расширение:)fenced_code_attributes . Как и в случае с изолированными блоками кода, можно использовать атрибуты в фигурных скобках или одно слово без скобок, которое будет рассматриваться как имя класса. Div заканчивается другой строкой, содержащей как минимум три последовательных двоеточия. Изолированный Div должен быть отделен пустыми строками от предыдущего и последующего блоков.

Пример:

::::: {#special .sidebar}
Here is a paragraph.

And another.
:::::

Изолированные блоки div могут быть вложенными. Открывающиеся заборы отличаются тем, что они должны иметь атрибуты:

::: Warning ::::::
This is a warning.

::: Danger
This is a warning within a warning.
:::
::::::::::::::::::

Заборы без атрибутов всегда закрывающие заборы. В отличие от кодовых блоков с ограждением, количество двоеточий в закрывающем ограждении не обязательно должно совпадать с числом в открывающемся ограждении. Однако для визуальной ясности может быть полезно использовать ограждения разной длины, чтобы отличать вложенные div от их родителей.

Расширение: bracketed_spans

Заключенная в скобки последовательность встроенных строк, которую можно было бы использовать для начала ссылки, будет рассматриваться как Spanс атрибутами, если за ней сразу же следуют атрибуты:

[This is *some text*]{.class key="val"}

Сноски

Расширение: footnotes

Pandoc Markdown позволяет использовать сноски, используя следующий синтаксис:

Here is a footnote reference,[^1] and another.[^longnote]

[^1]: Here is the footnote.

[^longnote]: Here's one with multiple blocks.

    Subsequent paragraphs are indented to show that they
belong to the previous footnote.

        { some.code }

    The whole paragraph can be indented, or just the first
    line.  In this way, multi-paragraph footnotes work like
    multi-paragraph list items.

This paragraph won't be part of the note, because it
isn't indented.

Идентификаторы в ссылках на сноски не могут содержать пробелов, табуляции или новых строк. Эти идентификаторы используются только для сопоставления ссылки на сноску с самим примечанием; в выводе сноски будут нумероваться последовательно.

Сами сноски не обязательно размещать в конце документа. Они могут появляться где угодно, кроме других элементов блока (списков, цитат, таблиц и т. Д.). Каждая сноска должна быть отделена от окружающего содержимого (включая другие сноски) пустыми строками.

Расширение: inline_notes

Также разрешены встроенные сноски (хотя, в отличие от обычных, они не могут содержать несколько абзацев). Синтаксис следующий:

Here is an inline note.^[Inlines notes are easier to write, since
you don't have to pick an identifier and move down to type the
note.]

Строчные и обычные сноски можно свободно смешивать.

(их pandoc-citeproc --bib2yamlможно получить из файла библиографии в одном из поддерживаемых форматов.)

Цитаты и ссылки могут быть отформатированы с использованием любого стиля, поддерживаемого языком стилей цитирования , перечисленных в репозитории стилей Zotero . Эти файлы указываются с помощью --cslпараметра или cslполя метаданных. По умолчанию pandoc-citeprocбудет использоваться формат даты автора Чикагского руководства по стилю . Проект CSL предоставляет дополнительную информацию о поиске и редактировании стилей .

Чтобы сделать ваши цитаты гиперссылками на соответствующие записи библиографии, добавьте link-citations: trueв свои метаданные YAML.

Цитаты заключаются в квадратные скобки и разделяются точкой с запятой. Каждая цитата должна иметь ключ, состоящий из '@' + идентификатора цитирования из базы данных, и может дополнительно иметь префикс, указатель и суффикс. Ключ цитирования должен начинаться с буквы, цифры или _, и может содержать буквенно-цифровые _символы, и внутренние знаки пунктуации ( :.#$%&-+?<>~/). Вот некоторые примеры:

Blah blah [see @doe99, pp. 33-35; also @smith04, chap. 1].

Blah blah [@doe99, pp. 33-35, 38-39 and *passim*].

Blah blah [@smith04; @doe99].

pandoc-citeprocобнаруживает термины локатора в файлах локали CSL . Принимаются как сокращенные, так и несокращенные формы. В en-USлокали термины локатора могут быть записаны в единственном или множественном числе, как book, bk./ bks.; chapter, chap./ chaps.; column, col./ cols.; figure, fig./ figs.; folio, fol./ fols.; number, no./ nos.; line, l./ ll.; note, n./ nn.; opus, op./ opp.; page, p./ pp.; paragraph, para./ paras.; part, pt./ pts.; section, sec./ secs.; sub verbo, s.v./ s.vv.;verse ,v. / vv.; volume, vol./ vols.; ¶/ ¶¶; §/ §§. Если термин локатора не используется, предполагается «страница».

pandoc-citeprocбудет использовать эвристику, чтобы отличить локатор от суффикса. В сложных случаях локатор можно заключить в фигурные скобки (используется только pandoc-citeproc0,15 и выше):

[@smith{ii, A, D-Z}, with a suffix]
[@smith, {pp. iv, vi-xi, (xv)-(xvii)} with suffix here]

Знак минус ( -) перед знаком @запретит упоминание автора в цитате. Это может быть полезно, когда автор уже упомянут в тексте:

Smith says blah [-@smith04].

Вы также можете написать цитату в тексте следующим образом:

@smith04 says blah.

@smith04 [p. 33] says blah.

Если для стиля требуется список цитируемых работ, он будет помещен в div с идентификатором refs, если он существует:

::: {#refs}
:::

В противном случае он будет помещен в конец документа. Создание библиографии можно запретить, установив suppress-bibliography: trueметаданные YAML.

Если вы хотите, чтобы в библиографии был заголовок раздела, вы можете установить его reference-section-titleв метаданных или поместить заголовок в начало div с идентификатором refs(если вы его используете) или в конце документа:

last paragraph...

# References

Библиография будет вставлена ​​после этого заголовка. Обратите внимание, что unnumberedкласс будет добавлен к этому заголовку, так что раздел не будет нумероваться.

Если вы хотите включить элементы в библиографию, не цитируя их в основном тексте, вы можете определить фиктивное nociteполе метаданных и поместить туда цитаты:

---
nocite: |
  @item1, @item2
...

@item3

В этом примере, документ будет содержать Цитирование item3только, но библиография будет содержать записи для item1, item2и item3.

Можно создать библиографию со всеми цитатами, независимо от того, появляются они в документе или нет, используя подстановочный знак:

---
nocite: |
  @*
...

Для вывода LaTeX вы также можете использовать natbibили biblatexдля отображения библиографии. Для этого укажите файлы библиографии, как описано выше, и добавьте аргумент --natbibили --biblatexаргумент для pandocвызова. Имейте в виду, что файлы библиографии должны быть в соответствующем формате (BibTeX или BibLaTeX).

Для получения дополнительной информации см. Справочную страницу pandoc-citeproc .

Расширения, не относящиеся к pandoc

Следующие расширения синтаксиса Markdown не включены по умолчанию в pandoc, но их можно включить, добавив +EXTENSIONк имени формата, где EXTENSION- имя расширения. Так, например, markdown+hard_line_breaksидет Markdown с жесткими переносами строк.

Расширение: attributes

Позволяет прикреплять атрибуты к любому встроенному или блочному элементу. Синтаксис атрибутов такой же, как в header_attributes.

  • Атрибуты, которые появляются сразу после встроенного элемента, влияют на этот элемент. Если они следуют за пробелом, значит, они принадлежат этому пространству. (Следовательно, этот параметр включает inline_code_attributesи link_attributes.)
  • Атрибуты, которые встречаются непосредственно перед блочным элементом, в отдельной строке, влияют на этот элемент.
  • Последовательные спецификаторы атрибутов могут использоваться как для блоков, так и для встроенных. Их атрибуты будут объединены.
  • Атрибуты, которые встречаются в конце текста заголовка Setext или ATX (отделенные пробелом от текста), влияют на элемент заголовка. (Следовательно, этот вариант включает header_attributes.)
  • Атрибуты, которые появляются после открывающегося ограждения в изолированном блоке кода, влияют на элемент блока кода. (Следовательно, этот вариант включает fenced_code_attributes.)
  • Атрибуты, которые встречаются в конце определения ссылочной ссылки, влияют на ссылки, которые ссылаются на это определение.

Обратите внимание, что AST pandoc в настоящее время не позволяет прикреплять атрибуты к произвольным элементам. Следовательно, при необходимости будет добавлен контейнер Span или Div.

Расширение: old_dashes

Выбирает поведение pandoc <= 1.8.2.1 для синтаксического анализа умных тире: -перед числовым значением является тире, а --перед ним - длинное тире. Этот параметр действует, только если smartон включен. Он автоматически выбирается для textileввода.

Расширение: angle_brackets_escapable

Разрешить <и >экранировать обратную косую черту, поскольку они могут быть в Markdown со вкусом GitHub, но не в исходном Markdown. Это подразумевается по умолчанию pandoc all_symbols_escapable.

Расширение: lists_without_preceding_blankline

Разрешить появление списка сразу после абзаца, без пробелов между ними.

Расширение: four_space_rule

Выбирает поведение pandoc <= 2.0 для синтаксического анализа списков, поэтому для абзацев продолжения элементов списка требуется отступ в четыре пробела.

Расширение: spaced_reference_links

Разрешите пробелы между двумя компонентами ссылочной ссылки, например,

[foo] [bar].

Расширение: hard_line_breaks

Заставляет все символы новой строки в абзаце интерпретироваться как разрывы строк вместо пробелов.

Расширение: ignore_line_breaks

Заставляет игнорировать новые строки в абзаце, а не рассматривать их как пробелы или жесткие разрывы строк. Этот параметр предназначен для использования с восточноазиатскими языками, где между словами не используются пробелы, а текст разделен на строки для удобства чтения.

Расширение: east_asian_line_breaks

Заставляет игнорировать новые строки в абзаце, а не рассматривать их как пробелы или жесткие разрывы строк, когда они встречаются между двумя широкими символами Восточной Азии. Это лучший выбор, чем ignore_line_breaksдля текстов, которые включают сочетание восточноазиатских символов и других символов.

Расширение: emoji

Разбирает текстовые смайлы, например :smile:смайлы Unicode.

Расширение: tex_math_single_backslash

Приводит к тому, что все, что находится между \(и, \)интерпретируется как встроенная математика TeX, а что-либо между \[и \]интерпретируется как математическое отображение TeX. Примечание: недостатком этого расширения является то, что оно не позволяет экранировать (и [.

Расширение: tex_math_double_backslash

Приводит к тому, что все, что находится между \\(и, \\)интерпретируется как встроенная математика TeX, а что-либо между \\[и \\]интерпретируется как математическое отображение TeX.

Расширение: markdown_attribute

По умолчанию pandoc интерпретирует материал внутри тегов уровня блока как Markdown. Это расширение изменяет поведение, так что Markdown анализируется только внутри тегов уровня блока, если теги имеют атрибут markdown=1.

Расширение: mmd_title_block

Включает основную надпись в стиле MultiMarkdown в верхней части документа, например:

Title:   My title
Author:  John Doe
Date:    September 1, 2008
Comment: This is a sample mmd title block, with
         a field spanning multiple lines.

Подробности см. В документации MultiMarkdown. Если pandoc_title_blockили yaml_metadata_blockвключен, он будет иметь приоритет mmd_title_block.

Расширение: abbreviations

Разбирает ключи сокращений PHP Markdown Extra, например

*[HTML]: Hypertext Markup Language

Обратите внимание, что модель документа pandoc не поддерживает аббревиатуры, поэтому, если это расширение включено, ключи сокращений просто пропускаются (в отличие от анализа как абзацев).

Расширение: autolink_bare_uris

Превращает все абсолютные URI в ссылки, даже если они не заключены в фигурные скобки <...>.

Расширение: mmd_link_attributes

Анализирует атрибуты "ключ-значение" в стиле мультимаркинга в ссылках и ссылках на изображения. Это расширение не следует путать с link_attributesрасширением.

This is a reference ![image][ref] with multimarkdown attributes.

[ref]: https://path.to/image "Image title" width=20px height=30px
       id=myId class="myClass1 myClass2"

Расширение: mmd_header_identifiers

Анализирует идентификаторы заголовков в стиле multitimarkdown (в квадратных скобках, после заголовка, но перед любыми конечными #s в заголовке ATX).

Расширение: compact_definition_lists

Активирует синтаксис списка определений pandoc 1.12.x и более ранних версий. Этот синтаксис отличается от синтаксиса, описанного выше в разделе Списки определений, в нескольких отношениях:

  • Между последовательными элементами списка определений не требуется пустых строк.
  • Чтобы получить «плотный» или «компактный» список, опускайте пробелы между последовательными элементами; пробел между термином и его определением ни на что не влияет.
  • Ленивый перенос абзацев не допускается: все определение должно иметь отступ в четыре пробела. 4

Расширение: gutenberg

Используйте для вывода соглашения Project Gutenbergplain : заглавные буквы для сильного выделения, окружение подчеркиванием для регулярного выделения, добавление дополнительных пробелов вокруг заголовков.

Создание слайд-шоу с помощью pandoc

Вы можете использовать pandoc для создания слайд-презентации HTML + JavaScript, которую можно будет просматривать через веб-браузер. Есть пять способов сделать это, используя S5 , DZSlides , Slidy , Slideous или reveal.js . Вы также можете создавать слайд-шоу PDF с помощью LaTeX beamerили слайд-шоу в формате Microsoft PowerPoint .

Вот исходный код Markdown для простого слайд-шоу habits.txt:

% Habits
% John Doe
% March 22, 2005

# In the morning

## Getting up

- Turn off alarm
- Get out of bed

## Breakfast

- Eat eggs
- Drink coffee

# In the evening

## Dinner

- Eat spaghetti
- Drink wine

------------------

![picture of spaghetti](images/spaghetti.jpg)

## Going to sleep

- Get in bed
- Count sheep

Чтобы создать слайд-шоу HTML / JavaScript, просто введите

pandoc -t FORMAT -s habits.txt -o habits.html

где FORMATлибо s5, slidy, slideous, dzslides, или revealjs.

Для Slidy, Slideous, discover.js и S5 файл, созданный pandoc с -s/--standaloneопцией, включает ссылку на файлы JavaScript и CSS, которые, как предполагается, доступны по относительному пути s5/default(для S5), slideous(для Slideous), reveal.js( для раскрытия.js) или на сайте Slidy по адресу w3.org(для Slidy). (Эти пути могут быть изменены путем установки slidy-url, slideous-url, revealjs-urlили s5-urlпеременных, см переменных для HTML горок . Выше) Для DZSlides, (относительно короткий) JavaScript и CSS включены в файл по умолчанию.

Для всех форматов слайдов HTML эту --self-containedопцию можно использовать для создания единого файла, который содержит все данные, необходимые для отображения слайд-шоу, включая связанные сценарии, таблицы стилей, изображения и видео.

Чтобы создать слайд-шоу PDF с помощью проектора, введите

pandoc -t beamer habits.txt -o habits.pdf

Обратите внимание, что слайд-шоу Show.js также можно преобразовать в PDF, распечатав его в файл из браузера.

Чтобы создать слайд-шоу Powerpoint, введите

pandoc habits.txt -o habits.pptx

Структурирование слайд-шоу

По умолчанию уровень слайда - это самый высокий уровень заголовка в иерархии, за которым сразу следует содержимое, а не другой заголовок где-нибудь в документе. В приведенном выше примере за заголовками уровня 1 всегда следует заголовок уровня 2, за которым следует контент, поэтому уровень слайда равен 2. Это значение по умолчанию можно изменить с помощью --slide-levelпараметра.

Документ разбивается на слайды по следующим правилам:

  • Горизонтальная линейка всегда начинает новый слайд.

  • Заголовок на уровне слайда всегда начинает новый слайд.

  • Заголовки ниже уровня слайда в иерархии создают заголовки внутри слайда.

  • Заголовки выше уровня слайда в иерархии создают «слайды заголовков», которые просто содержат заголовок раздела и помогают разбить слайд-шоу на разделы. Неслайд-контент под этими заголовками будет включен в титульный слайд (для слайд-шоу HTML) или в следующий слайд с тем же заголовком (для проектор).

  • Титульный лист создается автоматически из основной надписи документа, если таковая имеется. (В случае проектора это можно отключить, закомментировав некоторые строки в шаблоне по умолчанию.)

Эти правила предназначены для поддержки множества разных стилей слайд-шоу. Если вас не волнует структурирование слайдов на разделы и подразделы, вы можете просто использовать заголовки уровня 1 для всех слайдов. (В этом случае уровень 1 будет уровнем слайда.) Но вы также можете разделить слайд-шоу на разделы, как в примере выше.

Примечание: в слайд-шоу Show.js, если уровень слайда равен 2, будет создан двухмерный макет с заголовками уровня 1, построенными по горизонтали, и заголовками уровня 2, построенными по вертикали. Не рекомендуется использовать более глубокое вложение уровней разделов с помощью detect.js.

Дополнительные списки

По умолчанию эти авторы создают списки, отображающие «все сразу». Если вы хотите, чтобы ваши списки отображались постепенно (по одному элементу за раз), используйте эту -iопцию. Если вы хотите, чтобы конкретный список отличался от значения по умолчанию, поместите его в divблок с классом incrementalили nonincremental. Так, например, используя fenced divсинтаксис, следующее будет инкрементным независимо от значения документа по умолчанию:

::: incremental

- Eat spaghetti
- Drink wine

:::

или

::: nonincremental

- Eat spaghetti
- Drink wine

:::

Хотя использование incrementalи nonincrementaldiv являются рекомендуемым методом настройки инкрементных списков для каждого случая, также поддерживается более старый метод: размещение списков внутри цитаты будет отличаться от документа по умолчанию (то есть, он будет отображаться постепенно без -iопции и все сразу с -iопцией):

> - Eat spaghetti
> - Drink wine

Оба метода позволяют смешивать инкрементные и не инкрементные списки в одном документе.

Примечание. Ни этот -i/--incrementalвариант, ни какой-либо из описанных здесь методов в настоящее время не работают для вывода PowerPoint.

Вставка пауз

Вы можете добавить «паузы» в слайд, включив абзац, содержащий три точки, разделенные пробелами:

# Slide with a pause

content before the pause

. . .

content after the pause

Примечание: эта функция еще не реализована для вывода PowerPoint.

Подсветка синтаксиса

Pandoc автоматически выделит синтаксис в изолированных блоках кода , которые отмечены названием языка. Библиотека Haskell skylighting используется для подсветки. В настоящее время выделение поддерживается только для вывода HTML, EPUB, Docx, Ms и LaTeX / PDF. Чтобы увидеть список языков, которые распознает pandoc, введите pandoc --list-highlight-languages.

Цветовую схему можно выбрать с помощью --highlight-styleопции. Цветовая схема по умолчанию - это pygments, имитирующая цветовую схему по умолчанию, используемую пигментами библиотеки Python (хотя пигменты фактически не используются для выделения). Чтобы увидеть список стилей выделения, введите pandoc --list-highlight-styles.

Если вас не устраивают предопределенные стили, вы можете использовать его --print-highlight-styleдля создания .themeфайла JSON, который можно изменить и использовать в качестве аргумента для --highlight-style. Чтобы получить версию pygmentsстиля в формате JSON , например:

pandoc --print-highlight-style pygments > my.theme

Затем отредактируйте my.themeи используйте его так:

pandoc --highlight-style my.theme

Если вас не устраивает встроенная подсветка или вы хотите выделить язык, который не поддерживается, вы можете использовать --syntax-definitionопцию для загрузки файла определения синтаксиса XML в стиле KDE . Прежде чем писать свой собственный, взгляните на репозиторий определений синтаксиса KDE .

Чтобы отключить выделение, используйте --no-highlightопцию.

Пользовательские стили

Пользовательские стили можно использовать в форматах docx и ICML.

Вывод

По умолчанию вывод pandoc docx и ICML применяет предопределенный набор стилей для блоков, таких как абзацы и кавычки, и в основном использует форматирование по умолчанию (курсив, полужирный шрифт) для строк. Это будет работать для большинства целей, особенно с reference.docxфайлом. Однако, если вам нужно применить свои собственные стили к блокам или сопоставить уже существующий набор стилей, pandoc позволяет вам определять собственные стили для блоков и текста, используя divs и spans соответственно.

Если вы определяете a divили spanс атрибутом custom-style, pandoc будет применять указанный стиль к содержащимся в нем элементам (за исключением элементов, функция которых зависит от стиля, таких как заголовки, блоки кода, кавычки блоков или ссылки). Так, например, используя bracketed_spansсинтаксис,

[Get out]{custom-style="Emphatically"}, he said.

создаст файл docx со стилем «Выходи» со стилем символа Emphatically. Аналогично, используя fenced_divsсинтаксис,

Dickinson starts the poem simply:

::: {custom-style="Poetry"}
| A Bird came down the Walk---
| He did not know I saw---
:::

стилизовал бы две содержащиеся строки со Poetryстилем абзаца.

Для вывода docx стили будут определены в файле вывода как унаследованные от обычного текста, если стили еще не находятся в вашем reference.docx. Если они уже определены, pandoc не изменит определение.

Эта функция обеспечивает максимальную настройку в сочетании с фильтрами pandoc . Если вы хотите, чтобы все абзацы после кавычек были с отступом, вы можете написать фильтр для применения необходимых стилей. Если вы хотите, чтобы весь курсив был преобразован в Emphasisстиль символа (возможно, чтобы изменить их цвет), вы можете написать фильтр, который преобразует все выделенные курсивом встроенные строки в встроенные строки в рамках Emphasisпользовательского стиля span.

Для вывода docx вам не нужно включать какие-либо расширения для работы пользовательских стилей.

вход

Программа чтения docx по умолчанию считывает только те стили, которые она может преобразовать в элементы pandoc, либо путем прямого преобразования, либо путем интерпретации производных стилей входного документа.

Включив stylesрасширение в docx reader ( -f docx+styles), вы можете производить вывод, который поддерживает стили входного документа, используя custom-styleкласс. Стили абзацев интерпретируются как блоки div, а стили символов - как интервалы.

Например, используя custom-style-reference.docxфайл в каталоге test, мы получаем следующие разные результаты:

Без +stylesрасширения:

$ pandoc test/docx/custom-style-reference.docx -f docx -t markdown
This is some text.

This is text with an *emphasized* text style. And this is text with a
**strengthened** text style.

> Here is a styled paragraph that inherits from Block Text.

И с расширением:

$ pandoc test/docx/custom-style-reference.docx -f docx+styles -t markdown

::: {custom-style="First Paragraph"}
This is some text.
:::

::: {custom-style="Body Text"}
This is text with an [emphasized]{custom-style="Emphatic"} text style.
And this is text with a [strengthened]{custom-style="Strengthened"}
text style.
:::

::: {custom-style="My Block Style"}
> Here is a styled paragraph that inherits from Block Text.
:::

С помощью этих настраиваемых стилей вы можете использовать свой входной документ в качестве справочного документа при создании вывода docx (см. Ниже) и поддерживать те же стили во входных и выходных файлах.

Пользовательские писатели

Pandoc может быть расширен пользовательскими писателями, написанными на Lua . (Pandoc включает интерпретатор Lua, поэтому Lua не нужно устанавливать отдельно.)

Чтобы использовать настраиваемый модуль записи, просто укажите путь к сценарию Lua вместо формата вывода. Например:

pandoc -t data/sample.lua

Создание пользовательского писателя требует написания функции Lua для каждого возможного элемента в документе pandoc. Чтобы получить документированный пример, который вы можете изменить в соответствии с вашими потребностями, выполните

pandoc --print-default-data-file sample.lua

Примечание о безопасности

Если вы используете pandoc для преобразования пользовательского контента в веб-приложении, помните о следующих моментах:

  1. Хотя сам pandoc не будет создавать и изменять какие-либо файлы, кроме тех, которые вы явно запрашиваете у него (за исключением временных файлов, используемых при создании PDF-файлов), фильтр или пользовательский писатель в принципе могут делать что угодно в вашей файловой системе. Пожалуйста, внимательно проверьте фильтры и настраиваемые средства записи перед их использованием.

  2. Если ваше приложение использует pandoc как библиотеку Haskell (вместо того, чтобы использовать оболочку для исполняемого файла), можно использовать его в режиме, который полностью изолирует pandoc от вашей файловой системы, запустив операции pandoc в PandocPureмонаде. См. Документ Использование pandoc API для получения дополнительных сведений.

  3. Парсеры Pandoc могут показывать патологическую производительность в некоторых случаях. Целесообразно поставить любые операции pandoc на тайм-аут, чтобы избежать атак DOS, использующих эти проблемы. Если вы используете исполняемый файл pandoc, вы можете добавить параметры командной строки +RTS -M512M -RTS(например), чтобы ограничить размер кучи до 512 МБ.

  4. HTML, созданный pandoc, не может быть безопасным. Если этот параметр raw_htmlвключен для ввода Markdown, пользователи могут вводить произвольный HTML. Даже если raw_htmlон отключен, пользователи могут включать опасное содержимое в атрибуты заголовков, интервалов и блоков кода. На всякий случай следует запустить весь сгенерированный HTML-код через средство очистки HTML.
